## 1. 文档目的与使用约定

本索引是 22 份页面级 PRD 的公共契约，统一导航名称、角色、数据范围、对象术语、状态和通用产品规则。页面 PRD 不得重新定义本文件已定义的术语、状态或规则；如页面存在更严格的字段校验，应在该页面 PRD 中说明，且不得放宽本文件的约束。

新增、编辑、详情、审核、抽屉和弹窗均作为所属二级页面的子页面或子流程，不单独建立 PRD。页面级 PRD 按统一的 16 章结构编写；不适用的章节必须写明“不适用及原因”。

## 3. 角色与数据范围

角色缩写仅用于表格、字段说明和验收条件；页面展示使用完整角色名称。权限由菜单权限、操作权限、数据权限和字段权限共同决定，并遵循最小权限原则。

| 缩写 | 角色 | 主要职责 | 默认数据范围 |
|---|---|---|---|
| ADM | 平台管理员 | 账号、角色、城市商圈和系统配置管理；全局运营与审计 | 全部城市、全部团队、全部对象 |
| OPS | 运营人员 | 楼盘、房源、审核、举报和供给数据维护 | 获授权城市的楼盘、房源、商户与相关线索 |
| MGR | 销售主管 | 团队管理、线索分配、团队跟进质量管理 | 所属团队及获授权城市的线索、客户和跟进记录 |
| BRK | 经纪人 | 客户跟进、需求确认和房源推荐 | 本人负责的线索、客户和跟进记录；可查询获授权的上架房源 |
| CSR | 客服 | 基础咨询承接、线索新增和资料补充 | 获授权城市的咨询线索；不取得团队管理或审核权限 |

手机号等敏感字段默认脱敏。经纪人本人、销售主管和平台管理员可按字段权限查看其职责范围内的完整手机号；导出敏感字段需要独立权限。

## 4. 统一术语与状态

| 术语 | 统一定义 |
|---|---|
| 楼盘（Building） | 可被房源关联的办公物业或园区主数据，承载名称、城市、区域、商圈、地址、坐标和楼宇参数等稳定信息。楼盘不是具体可出租单元。 |
| 房源（Listing） | 隶属于一个楼盘、可被审核和展示的具体可租或可售空间单元，承载面积、价格、装修、可用日期、联系人和展示状态。 |
| 商户（Merchant） | 房源供给侧主体，包括业主、代理商、联合办公品牌和渠道方；MVP 仅管理基础资料与供给归属，不提供完整入驻流程。 |
| 经纪人（Broker） | 负责承接、跟进和推荐房源的业务人员，归属一个主团队并配置服务城市或商圈。 |
| 客户（Customer） | 由姓名、手机号、公司等信息识别的需求主体；一个客户可保留多个需求或线索历史。 |
| 线索（Lead） | 一次可分配、可跟进的租赁或出售需求记录，关联客户、来源、意向区域、面积、预算、负责人和阶段。 |
| 租赁 | 以周期性租金或单价表达的办公空间使用业务。金额必须保存数值、币种、计价周期和单位。 |
| 出售 | 以总价或单价表达的空间出售意向。MVP 仅保留可配置字段和展示、筛选、权限口径，不建设交易闭环。 |
| 直租 | 经确认由业主、资产方或其授权主体直接出租的房源属性；不等同于“免佣”，应独立维护。 |
| 直售 | 经确认由业主、资产方或其授权主体直接出售的房源属性；不等同于“直租”，应独立维护。 |
| 发布状态 | 面向前台可见性的房源状态：草稿、已上架、已下架、已出租。草稿、已下架和已出租均不可前台展示；已出租时必须自动下架。 |
| 审核状态 | 审核任务状态：待审核、审核通过、已驳回。草稿未提交审核时不创建审核任务；已驳回的房源须修改后重新提交。MVP 默认审核通过后直接上架。 |

房源的发布状态与审核状态是两个独立字段。页面可为便于阅读展示组合状态，但不得将“已驳回”作为发布状态，也不得让未审核通过的房源上架。线索阶段统一为：新线索、已分配、已联系、需求确认、已推荐、已流失、无效。

## 5. 通用产品与数据规则

| 主题 | 统一规则 |
|---|---|
| 时间 | 统一使用北京时间。列表显示到分钟；操作日志和导出保留到秒。页面涉及“今日”或时间范围时，按北京时间自然日计算。 |
| 金额与价格 | 金额、租金单价和出售价格必须同时保存数值、币种、计价周期和单位；展示层不得只保存拼接文本。租赁与出售的价格单位、总价和权限口径必须分开定义。 |
| 面积 | 以平方米为基础单位，支持一位小数。统计使用原始值，页面按展示规则格式化；区间筛选的下限不得大于上限。 |
| 内部 ID | 楼盘、房源、客户、商户和账号使用不可变内部 ID。名称、手机号、地址和登录账号不得替代业务主键。 |
| 去重 | 楼盘在同一城市内名称不可重复；客户或线索录入时按手机号查重，命中后提供合并到已有客户、创建新需求或取消的处理路径。 |
| 删除与停用 | 默认停用或逻辑删除。存在业务关联的数据不得物理删除；已被使用的城市、区域、商圈、角色和字典项只能停用，历史数据继续可查。跟进记录和审核记录不可物理删除。 |
| 导出 | 导出继承当前查询条件和数据权限，敏感字段按角色脱敏或受字段权限控制，并记录导出日志。线索导入须预校验，重复手机号默认不导入且输出失败原因。 |
| 批量操作 | 必须显示影响数量、要求二次确认、返回逐条结果及失败原因。批量分配仅允许具备分配权限的角色执行。 |
| 并发编辑 | 采用版本号或更新时间检测。提交旧版本时阻止覆盖，提示刷新后再处理；审核中房源不可再次提交，需撤回后编辑。 |
| 审计与反馈 | 新增、编辑、删除或停用、审核、上下架、分配、导出和权限变更必须记录操作日志。危险操作二次确认；驳回、取消、流失、无效等负向操作必须填写原因。 |
| 列表与异常 | 列表支持筛选、重置、分页、排序和空状态，默认每页 20 条并支持 20、50、100 条。查询失败提供错误提示和重试；详情返回来源列表时保留筛选条件。 |

## 7. MVP 排除范围

本次页面 PRD 不引入在线支付、佣金分账、发票、财务结算、电子签章、在线合同、成交报备、合同归档、预约带看完整闭环、电话外呼与录音、IM 深度集成、VR 看房制作、智能推荐算法、自动派单策略引擎、复杂风控模型、完整商户开放平台或复杂 CMS 专题搭建器。

## 8. 阅读与评审指引

1. 先阅读本索引，确认术语、角色、状态和公共规则。
2. 再按业务依赖阅读楼盘管理、房源管理、客户线索、商户经纪、工作台与数据看板、系统设置。
3. 每份页面 PRD 必须使用本索引的角色缩写、状态名称和数据口径，并在“截图参考”章节回链本文件第 6 节对应能力。
4. 评审时按正常流程、权限边界、异常恢复、数据正确性四类验收条件检查；指标须说明业务定义、计算公式、去重口径、时间口径、刷新频率和下钻目标。
5. 发现跨页术语、状态、字段口径或数据范围冲突时，以本索引为准，并在变更本索引后同步评审受影响页面 PRD。
