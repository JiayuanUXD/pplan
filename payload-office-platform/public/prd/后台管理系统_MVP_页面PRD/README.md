## 1. 文档目的与使用约定

本索引是 22 份页面级 PRD 的公共契约，统一导航名称、角色、数据范围、对象术语、状态和通用产品规则。页面 PRD 不得重新定义本文件已定义的术语、状态或规则；如页面存在更严格的字段校验，应在该页面 PRD 中说明，且不得放宽本文件的约束。

本目录内的 README 与 22 份页面级 PRD 是 MVP 当前唯一交付基线，并替代上述两份旧版汇总 PRD 的页面规则。旧版文档仅保留为需求演进记录，不再作为设计、研发、测试或验收依据；发生冲突时必须以本索引及本目录页面 PRD 为准。

新增、编辑、详情、审核、抽屉和弹窗均作为所属二级页面的子页面或子流程，不单独建立 PRD。页面级 PRD 按统一的 16 章结构编写；不适用的章节必须写明“不适用及原因”。

## 3. 角色与数据范围

角色缩写仅用于表格、字段说明和验收条件；页面展示使用完整角色名称。权限由菜单权限、操作权限、数据权限和字段权限共同决定，并遵循最小权限原则。

| 缩写 | 角色 | 主要职责 | 默认数据范围 |
|---|---|---|---|
| ADM | 平台管理员 | 账号、角色、城市商圈和系统配置管理；全局运营与审计 | 全部城市、全部团队、全部对象 |
| OPS | 运营人员 | 楼盘、房源、审核、举报和供给数据维护 | 获授权城市的楼盘、房源、商户与相关线索 |
| MGR | 销售主管 | 团队管理、线索分配、团队跟进质量管理 | 所属团队及获授权城市的线索、客户和跟进记录 |
| BRK | 经纪人 | 客户跟进、需求确认和房源推荐 | 本人负责的线索、客户和跟进记录；可查询获授权的上架房源 |
| CSR | 客服 | 基础咨询承接、线索新增和资料补充 | 获授权城市的咨询线索；不取得团队管理或审核权限 |

手机号等敏感字段默认脱敏。经纪人本人、销售主管和平台管理员可按字段权限查看其职责范围内的完整手机号；导出敏感字段需要独立权限。

## 4. 统一术语与状态

| 术语 | 统一定义 |
|---|---|
| 楼盘（Building） | 可被房源关联的办公物业或园区主数据，承载名称、城市、区域、商圈、地址、坐标和楼宇参数等稳定信息。楼盘不是具体可出租单元。 |
| 房源（Listing） | 隶属于一个楼盘、可被审核和展示的具体可租或可售空间单元，承载面积、价格、装修、可用日期、联系人和展示状态。 |
| 商户（Merchant） | 房源供给侧主体，固定 `MERCHANT_TYPE` 为 `OWNER`（业主）、`AGENCY`（代理商）、`FLEX_OFFICE_BRAND`（联合办公品牌）和 `CHANNEL`（渠道方）；MVP 仅管理基础资料与供给归属，不提供完整入驻流程。 |
| 经纪人（Broker） | 负责承接、跟进和推荐房源的业务人员，归属一个主团队并配置服务城市或商圈。 |
| 客户（Customer） | 由姓名、手机号、公司等信息识别的需求主体；一个客户可保留多个需求或线索历史。 |
| 线索（Lead） | 一次可分配、可跟进的租赁或出售需求记录，关联客户、来源、意向区域、面积、预算、负责人和阶段。 |
| 租赁 | 以周期性租金或单价表达的办公空间使用业务。金额必须保存数值、币种、计价周期和单位。 |
| 出售 | 以总价或单价表达的空间出售意向。MVP 仅保留可配置字段和展示、筛选、权限口径，不建设交易闭环。 |
| 直租 | 经确认由业主、资产方或其授权主体直接出租的房源属性；不等同于“免佣”，应独立维护。 |
| 直售 | 经确认由业主、资产方或其授权主体直接出售的房源属性；不等同于“直租”，应独立维护。 |
| 发布状态 | 面向前台可见性的房源状态：草稿、已上架、已下架、已出租。草稿、已下架和已出租均不可前台展示；已出租时必须自动下架。 |
| 审核状态 | 房源审核结论状态：未提交、待审核、审核通过、已驳回。未提交时不创建审核任务；已驳回的房源须修改后重新提交。审核队列的领取与处理进度使用独立任务状态，不得混入本字段。审核通过默认只写入本字段，不隐式改变发布状态。 |

房源的发布状态与审核状态是两个独立字段。页面可为便于阅读展示组合状态，但不得将“已驳回”作为发布状态，也不得让未审核通过的房源上架。审核通过只更新 `review_status`；仅同一操作人在提交审核结论时同时具备审核和发布权限、明确勾选“通过后上架”，并满足有效供给谓词时，才可在同一原子事务将 `review_status=审核通过` 与 `publication_status=已上架` 一并写入。其他情形保持草稿或已下架，由具备发布权限者另行显式上架。线索阶段统一为：`新建 -> 待分配 -> 跟进中 -> 有效商机 -> 带看 -> 谈判 -> 已转化/已流失`；公海表示归属状态，不是线索阶段；合并来源行使用独立合并状态，不得加入生命周期。

## 5. 通用产品与数据规则

| 主题 | 统一规则 |
|---|---|
| 时间 | 统一使用北京时间。列表显示到分钟；操作日志和导出保留到秒。页面涉及“今日”或时间范围时，按北京时间自然日计算。 |
| 金额与价格 | 金额、租金单价和出售价格必须同时保存数值、币种、计价周期和单位；展示层不得只保存拼接文本。租赁与出售的价格单位、总价和权限口径必须分开定义。 |
| 面积 | 以平方米为基础单位，支持一位小数。统计使用原始值，页面按展示规则格式化；区间筛选的下限不得大于上限。 |
| 内部 ID | 楼盘、房源、客户、商户和账号使用不可变内部 ID。名称、手机号、地址和登录账号不得替代业务主键。 |
| 去重 | 楼盘在同一城市内名称不可重复；客户或线索录入时按手机号查重，命中后提供合并到已有客户、创建新需求或取消的处理路径。 |
| 删除与停用 | 默认停用或逻辑删除。存在业务关联的数据不得物理删除；已被使用的城市、区域、商圈、角色和字典项只能停用，历史数据继续可查。跟进记录和审核记录不可物理删除。 |
| 导出 | 导出继承当前查询条件和数据权限，敏感字段按角色脱敏或受字段权限控制，并记录导出日志。线索导入须预校验，重复手机号默认不导入且输出失败原因。 |
| 批量操作 | 必须显示影响数量、要求二次确认、返回逐条结果及失败原因。批量分配仅允许具备分配权限的角色执行。 |
| 供给关系有效期 | Building 默认商户关系与 Listing 商户关系统一使用半开区间 `[effective_from,effective_to)`，`effective_to` 可空；非空时必须 `effective_from < effective_to`。数据库分别以 `building_id`、`listing_id` 的区间排斥约束禁止重叠（空结束视为无穷），任一时点最多一个有效关系。边界时刻归属新关系：`t=effective_to` 不属于旧关系，若新关系从该时刻开始则由其确定；Listing 在其关系 `effective_from` 继承该时点有效的 Building 默认商户快照，后续 Building 关系变更不得回写既有 Listing 关系。 |
| 有效供给谓词 | 前台、楼盘聚合、咨询/顾问候选和有效供给看板必须复用同一谓词：Listing 未逻辑删除、`publication_status=已上架`、`review_status=审核通过`、`supply_visibility_hold=正常`、未被举报暂停且媒体完整；Building、其所属城市和区域均启用；当前 Listing 商户关系有效，商户启用、资质已通过且未过期，且商户服务城市已启用并覆盖 Building 所属城市。任一条件不满足即排除，禁止各页面自行放宽。 |
| 商户停用与复核冻结 | 商户停用批次必须在同一事务/可靠编排中把全部关联 Listing 持久化为 `supply_visibility_hold=待复核`，保留审核和发布状态及关系历史。商户恢复不得自动清除该值；运营人员仅在商户、Building 所属城市/区域、商户服务城市、当前供给关系和审核结论均有效后，才可显式解除为“正常”，随后仍须由具备发布权限者显式上架。停用、解除、权限校验与前后值均须审计。 |
| 并发编辑 | 采用版本号或更新时间检测。提交旧版本时阻止覆盖，提示刷新后再处理；审核中房源不可再次提交，需撤回后编辑。 |
| 审计与反馈 | 新增、编辑、删除或停用、审核、上下架、分配、导出和权限变更必须记录操作日志。危险操作二次确认；驳回、取消、流失、无效等负向操作必须填写原因。 |
| 列表与异常 | 列表支持筛选、重置、分页、排序和空状态，默认每页 20 条并支持 20、50、100 条。查询失败提供错误提示和重试；详情返回来源列表时保留筛选条件。 |

### 5.1 MVP-R1 运行参数发布基线

下表是本期线索分配、认领、跟进与回收共用的发布基线，不是后台可配置项。每次分配、认领、SLA 实例、回收判断和跟进纠错都必须持久化 `runtime_policy_version=MVP-R1`，以及实际命中的 seconds/limits 值；不得只依赖当时的默认值。任何数值调整必须发布新的运行参数版本并完成变更评审，不得通过本期 UI、字典配置或角色配置修改。

| 参数 | MVP-R1 值 | 持久化字段和值 |
|---|---:|---|
| 分配时限 | 2 小时 | `assignment_due_seconds=7200` |
| 首次有效跟进时限 | 4 小时 | `first_valid_follow_up_due_seconds=14400` |
| 认领保护期 | 24 小时 | `claim_protection_seconds=86400` |
| 无有效跟进回收时限 | 72 小时 | `no_valid_follow_up_recovery_seconds=259200` |
| 每日成功认领上限 | 20 条 | `daily_claim_limit=20` |
| 活跃自有线索上限 | 100 条 | `active_owned_lead_limit=100` |
| 跟进纠错窗口 | 24 小时 | `follow_up_correction_seconds=86400` |
