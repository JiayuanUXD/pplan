## 2. 页面概述

经纪人是归属一个启用商户、完成身份与任职核验的个人，可加入一个主团队并负责线索跟进。页面维护其身份、任职、服务范围、专业能力、承接容量、前台展示和联系方式控制，支撑楼盘/房源顾问归属与线索路由。

页面成功标准是：新增线索只会分配给在服务范围内且有剩余容量的合格经纪人；前台只展示真实、可联系、与当前楼盘或房源存在有效关系的顾问。MVP 不包含佣金结算、排班、智能评分或跨平台社交账号运营。

## 3. 用户与权限

| 角色 | 数据范围与权限 |
|---|---|
| 平台管理员 | 全量查看、编辑、核验、启停用、配置展示与联系权限、导出；可填写覆盖原因后跨团队执行线索转派或回收。 |
| 运营人员 | 仅授权城市；新增、编辑、提交核验、查看关系；不能执行线索转派/回收、跨城市操作或授予完整联系方式导出。 |
| 销售主管 | 本团队及授权服务范围；查看成员、执行本团队非终态线索转派/回收和查看脱敏联系方式；不可核验身份、修改商户、修改活跃自有线索上限或越权启用。 |
| 经纪人 | 仅查看和编辑本人允许维护的头像、简介、专业标签、服务偏好；不能编辑核验、商户、团队、容量、展示与联系方式权限。 |

身份证号、身份证件、人脸/任职证明、手机号和二维码均为敏感信息。后台手机号默认脱敏；前台绝不显示身份证信息，联系人手机号是否显示仅由“联系方式展示权限”决定，不受后台查看权限替代。

## 4. 页面入口与关联

- 菜单入口：`商户经纪 > 经纪人管理`；商户详情、团队详情、楼盘/房源顾问选择器及线索详情可带参数进入。
- 前置条件：商户已建立，城市区域、专业标签字典、团队和账号基础数据可用。
- 上游来源：商户资质状态、账号状态、团队成员关系、楼盘/房源顾问关系、线索负责人和跟进记录。
- 下游去向：可跳转其关联商户、主团队、关联楼盘、关联房源、负责线索/客户、跟进记录及操作日志。
- 返回规则：子页返回保留筛选、排序、分页和 URL 状态；从线索转移抽屉关闭后回到发起页。

## 5. 页面结构

页面由顶部操作区、可折叠查询区、列表区和详情抽屉组成。详情页签顺序为：基础资料、身份与任职核验、服务与专业能力、团队与容量、前台展示与联系方式、关联楼盘/关联房源、负责线索与转移记录、审计。列表默认按状态、可分配性、更新时间排序；在窄屏下列表横向滚动，转移抽屉保持固定确认区。

## 8. 详情及表单

### 8.1 基础、核验与服务资料

| 字段 | 类型与校验 |
|---|---|
| 姓名/头像 | 姓名必填，2-50 字；头像可选，仅在核验通过且展示合格时用于前台。 |
| 手机号 | 必填，格式校验；同一活跃经纪人手机号唯一。 |
| 归属商户 | 必填，必须为已启用、资质已通过且未过期的商户。 |
| 身份证件 | 必填上传证件类型、证件号、有效期和附件；证件号加密存储并按权限查看。 |
| 任职证明 | 必填，包含商户、岗位、生效日期和证明附件；商户变更后必须重新核验。 |
| 服务城市/区域/商圈 | 至少一个启用服务城市；下级范围必须属于服务城市。 |
| 专业标签/简介 | 标签来自字典；简介 500 字内，禁止填写手机号、二维码或外部联系方式。 |
| 主团队 | 启用经纪人必须有且仅有一个启用主团队；未分配状态不能承接新增线索。 |
| 活跃自有线索上限 | 只读展示命中的全局配置，默认每名经纪人 100 条非终态自有线索；由平台管理员统一维护，不按自然日重置，销售主管不得修改默认值或为成员单独覆盖。 |

### 8.2 前台展示、二维码与关联关系

前台展示开关、联系方式展示权限和二维码分别配置。二维码仅在派生状态为“前台展示”时生成，二维码内容只引用经纪人不可变 ID 和有效期，不内嵌手机号。关联楼盘/关联房源关系必须记录对象 ID、关系类型（招商顾问/房源顾问）、生效时间、失效时间和维护人；同一对象可以有多个顾问，但前台排序由楼盘/房源侧配置。

保存使用版本号校验；修改商户、团队、服务范围、容量或展示开关时，系统实时预览对线索分配、前台展示和关联对象的影响。

## 9. 核心操作与流程

### 9.1 核验、启用与可分配

1. 新建经纪人初始为草稿，完成身份和任职资料后提交核验。
2. 平台管理员分别核验身份和任职；任一驳回时不能启用，保留驳回原因。
3. 启用条件：归属商户启用且资质有效、身份核验通过且未过期、任职核验通过且有效、存在一个启用主团队、至少一个有效服务城市，且全局活跃自有线索上限配置有效。
4. 线索分配候选条件：经纪人启用、上述启用条件持续满足、主团队启用、服务范围覆盖线索意向城市/区域/商圈（线索未填下级范围时只校验城市）、当前非终态自有线索数小于活跃自有线索上限。该上限由平台管理员维护，默认 100 条，不按自然日重置；手动分配、销售主管操作和平台管理员覆盖均不得造成超限。

### 9.2 前台展示资格：精确判定

经纪人的前台展示状态仅为派生值“前台展示”或“隐藏”，不可手工覆盖。仅在以下全部条件同时满足时，状态才为“前台展示”，经纪人才作为前台楼盘或房源的“咨询顾问”出现，并显示其获授权的联系方式或有效二维码：

1. 经纪人业务状态为启用，归属商户业务状态为启用，且商户资质已通过、未过期。
2. 身份核验和任职核验均为已通过且在有效期内。
3. 主团队为启用状态，服务范围覆盖当前楼盘/房源所在城市及已配置的区域/商圈。
4. 前台展示开关为开，且联系方式展示权限为开；二维码如展示，必须未过期且未被失效。
5. 存在一条生效中的经纪人-楼盘或经纪人-房源关系，且该楼盘/房源自身为前台可见状态并满足其商户可发布条件。

任何一项不满足，派生状态即为“隐藏”：前台立即整体隐藏该顾问、联系方式、二维码入口和默认咨询路由；历史关联与历史咨询记录保留。联系方式展示权限或前台展示开关任一关闭时必须整体隐藏，不保留任何顾问条目或替代咨询状态。

### 9.3 停用影响与线索转移

停用经纪人必须填写原因并二次确认。系统锁定新分配、前台展示、二维码生成和新增顾问关联；若存在非终态线索，必须在停用前逐条或批量完成转派或回收，整批处置成功后才允许停用。非终态仅指新建、待分配、跟进中、有效商机、带看、谈判；终态仅指已转化、已流失：

1. 转入人必须满足线索分配候选条件，并且服务范围覆盖该线索；批量转移按每条线索校验容量。
2. 销售主管在所属团队执行全部非终态线索的转派或回收；平台管理员可填写覆盖原因后跨团队执行。直接转派后，负责人、所属团队快照和权限范围从生效时间起更新；原经纪人保留历史负责人、跟进和转移审计，不可再查看转出后线索的实时详情。
3. 无合格接收人时回收到公海，复用既有枚举 `public_pool_reason=销售主管回收`，并写上下文字段 `reason_detail=经纪人停用`，不得新增入池原因枚举。公海交接须在单个原子事务中清空 `owner_id`，追加不可变归属历史（原负责人、原团队、原商户、操作人、原因、时间），保留全部来源、归因、跟进、阶段、附件和操作历史，且绝不改变线索生命周期阶段。转派与回收覆盖上述六个非终态；已转化、已流失均不处理。
4. 停用生效时，所有有效楼盘/房源顾问关系保留为历史但立刻失效，前台顾问、联系方式、二维码和默认咨询路由立即移除；关联楼盘/房源不自动下架。

重新启用不会恢复已失效关系、二维码、前台展示和线索归属；必须重新核验条件、重新建立有效关联并显式开启相应开关。

## 10. 状态与业务规则

| 对象 | 枚举与规则 |
|---|---|
| 经纪人业务状态 | 草稿、核验中、启用、停用；启用与停用可互转，但重新启用须重新校验当前条件。 |
| 身份/任职核验 | 待提交、核验中、已通过、已驳回、已过期；任一非“已通过且有效”均不可启用或前台展示。 |
| 承接状态 | 可分配、已满、不可分配；由状态、商户、团队、服务范围和容量实时派生，不可手工覆盖。 |
| 前台展示状态 | 前台展示、隐藏；仅由第 9.2 节全部条件累计判定，任一条件不满足即隐藏。 |
| 团队关系 | 同一时间只能有一个主团队；成员转移保留起止时间，旧团队不保留实时数据权限。 |

经纪人不得物理删除。服务范围、商户或主团队变更不得使已有非终态线索失去责任主体；若新条件不覆盖已有线索，必须先按第 9.3 节转派或回收，禁止以历史负责人名义继续新增处理。活跃自有线索上限为平台管理员维护的全局配置，默认 100 条非终态自有线索且不按自然日重置，销售主管不可修改。账号停用、商户停用、团队停用或任一核验过期会即时重算承接和前台展示状态。

## 11. 异常与边界

- 查询和详情加载失败显示重试，保留当前 URL 状态；无数据展示空状态。
- 无完整手机号权限时不得通过错误提示、排序或导出反推出号码。
- 核验附件缺失、过期、病毒扫描失败或证件号重复时阻止提交并定位字段。
- 停用预览与确认之间若新增线索或容量变化，重新计算后要求重新确认。
- 转移中任一线索无合格接收人时，该线索可选进入公海；选择转派模式时整次操作失败，不产生部分转派。
- 楼盘/房源下架或商户停用时，展示资格异步重算不得超过 5 分钟；重算期间前台按保守策略隐藏顾问。

## 12. 数据与指标

数据来源为经纪人主数据、账号、商户资质、团队成员关系、城市区域、楼盘/房源关系、线索和跟进记录、二维码服务及审计日志。当前非终态自有线索数统计 `owner_id` 为该经纪人且阶段为新建、待分配、跟进中、有效商机、带看或谈判的去重线索；已转化、已流失不计入。活跃自有线索上限命中平台管理员维护的全局配置版本，默认 100 条，不按自然日重置。

前台展示资格和分配候选实时或事件驱动重算，缓存不超过 5 分钟；启停用、核验、关系、容量和上游状态变化须立即失效。导出继承数据权限和筛选条件，完整联系方式与证件字段仅对具备相应导出权限者提供。

## 13. 埋点、通知与审计

埋点包含查询、详情、核验提交/结论、展示开关、联系方式权限、二维码生成/失效、全局上限版本命中、顾问关系变更、线索转移预览/完成、启用和停用。核验驳回通知提交人；证件或任职证明到期前 30 天通知经纪人、商户主联系人与销售主管；停用前有待转移线索通知销售主管。

审计记录操作人、经纪人 ID、关联商户/团队 ID、前后状态、核验结论、前台展示派生结果及未满足条件、二维码标识、全局上限配置版本、转移线索 ID/接收人、`public_pool_reason`、`reason_detail`、原负责人/团队/商户快照、覆盖原因、时间（秒）和 IP（可选）。归属历史和审计记录不可修改或删除。

## 14. 验收标准

| 类别 | 可测试条件 |
|---|---|
| 正常 | 核验通过、商户和团队启用、范围覆盖且容量未满的经纪人可进入线索分配候选人。 |
| 正常 | 满足第 9.2 节五项条件的经纪人，在其有效关联且前台可见的楼盘/房源中显示为顾问。 |
| 权限 | 销售主管只能查看本团队成员且看不到证件与完整手机号；经纪人不能修改核验、团队、商户或联系方式权限。 |
| 异常 | 存在非终态线索时不能直接停用；转入人满载或范围不匹配时不得转派，并按既有“销售主管回收”原因交接公海。 |
| 数据 | 停用转派/回收覆盖全部六个非终态且不处理已转化、已流失；公海交接原子清空 `owner_id`、追加不可变归属历史并保持生命周期阶段不变。 |
| 数据 | 停用、核验过期、商户/团队停用、关闭展示或关系失效后，前台顾问和默认咨询路由被移除，历史跟进和关联可审计。 |
| 数据 | 重新启用经纪人不会自动恢复二维码、展示开关、顾问关系或转出的线索。 |

## 15. 依赖与风险

依赖商户管理、团队管理、账号与角色、城市区域、楼盘/房源、线索池/公海、文件存储、二维码和操作日志。风险包括历史经纪人缺少双核验或团队关系、服务范围粒度与线索需求不一致、容量设置不合理及前台资格事件延迟；上线前应完成历史数据补齐、资格回算、失效二维码清理和高并发转移演练。
