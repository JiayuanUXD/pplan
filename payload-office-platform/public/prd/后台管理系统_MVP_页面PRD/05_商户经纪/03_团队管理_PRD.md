## 2. 页面概述

团队是用于内部线索分配、管理层级和数据范围的组织分组，不是商户合同主体，也不改变经纪人与商户的任职关系。每个启用经纪人同一时间必须有一个主团队；团队主管通过团队获得成员线索和跟进记录的数据权限。

页面目标是维护可用的团队层级、主管、成员、服务范围和分配容量，使线索可在合规团队内分配并可追溯团队业绩。成功标准为：团队停用或删除前不存在未迁移的成员、主管或未完成线索；团队范围变化不会让现有线索失去可见和可处理的责任主体。MVP 不包含多级结算、虚拟团队、复杂矩阵汇报或自动业绩奖金。

## 3. 用户与权限

| 角色 | 数据范围与权限 |
|---|---|
| 平台管理员 | 全城市查看、建立层级、设置主管/成员/范围、启停用、逻辑删除、跨城市转移、导出。 |
| 城市运营 | 仅授权城市；新增、编辑、查看影响、发起成员与线索转移；不可跨城市转移或删除有历史关联团队。 |
| 销售主管 | 仅本团队；查看成员、线索、容量和汇总表现，可发起成员调整申请与团队内线索分配，不可更改团队层级、主管或数据范围。 |
| 经纪人、客服 | 无团队管理菜单；经纪人仅可查看本人主团队名称和服务范围。 |

团队列表和详情按团队数据权限过滤。销售主管可查看本团队及下级团队（如配置）成员的脱敏信息、线索和跟进记录，不可查看无授权团队数据；导出继承相同范围并记录审计。

## 4. 页面入口与关联

- 菜单入口：`商户经纪 > 团队管理`。
- 前置条件：城市区域、账号角色、商户、经纪人和线索数据可用。
- 上游来源：城市区域配置、账号/角色、经纪人主团队关系、商户归属、线索负责人及跟进记录。
- 下游去向：团队详情可跳转成员经纪人、团队线索、跟进记录、经营/线索看板；经纪人管理使用团队作为主团队候选；线索池以团队范围过滤和分配。
- 返回规则：跳转下游携带团队 ID，返回时保留团队树选择、查询、排序、分页和 URL 状态。

## 5. 页面结构

左侧为团队树，按父子层级展开当前用户有权节点；右侧从上至下为工具栏、查询区、列表/详情。详情页签包括基础与层级、主管、成员、服务范围与数据权限、线索分配与容量、转移记录、聚合表现和审计。新增或编辑使用抽屉；停用、转移和删除使用专用影响处置抽屉。窄屏时团队树折叠为选择器，成员与线索列表横向滚动。

## 6. 查询与筛选

| 控件 | 数据源与规则 |
|---|---|
| 关键词 | 团队名称、内部 ID、主管姓名/账号。 |
| 所属城市/服务范围 | 先选已启用城市，联动区域/商圈；只能选择权限范围内城市。 |
| 上级团队 | 仅可选同城市、启用且不构成循环引用的团队。 |
| 团队主管 | 仅显示有销售主管角色、账号启用且服务范围覆盖团队的经纪人/主管账号。 |
| 业务状态 | 草稿、启用、停用、已删除。默认不显示已删除。 |
| 成员与容量 | 成员数、可分配成员数、待转移成员、未完成线索数、超容量成员。 |
| 数据权限 | 本团队、含下级团队；仅平台管理员可修改。 |

默认按团队层级、排序号和更新时间展示。查询条件、树节点、页码和排序保留在 URL；重置恢复默认状态。已停用城市/区域仅可检索历史团队，不能作为新建或编辑值。

## 7. 列表或内容展示

| 字段 | 展示与规则 |
|---|---|
| 团队名称/内部 ID | 名称进入详情；内部 ID 不可变。 |
| 上级团队 | 顶级团队显示 `-`；已停用上级显示状态标签。 |
| 所属城市/服务范围 | 显示城市及最多两个下级范围，余项展示数量。 |
| 团队主管 | 显示姓名和账号状态；主管失效显示待处理。 |
| 成员数/可分配成员 | 成员为当前有效主团队关系；可分配成员满足经纪人分配候选条件。 |
| 未完成线索 | 负责人当前主团队为本团队且阶段未结束的去重线索数。 |
| 数据权限 | 本团队或含下级团队。 |
| 状态 | 草稿、启用、停用、已删除。 |
| 操作 | 查看、编辑、成员管理、线索分配、转移、启用、停用、删除。 |

支持按成员数、未完成线索数、创建/更新时间排序。无成员、无主管、无关联线索均显示 `0`；无数据展示空状态。列表每页默认 20 条，可切换 20/50/100。

## 8. 详情及表单

| 字段/分组 | 类型与校验 |
|---|---|
| 团队名称 | 必填，2-100 字；同一父团队下名称唯一，名称不作为关系主键。 |
| 所属城市 | 必填且只能一个启用城市；跨城市团队不支持。 |
| 上级团队 | 可选；必须同城市、启用且不形成自身或后代循环。 |
| 团队主管 | 启用团队必填；主管账号启用、有销售主管角色、经纪人核验和商户状态合格，且服务范围覆盖团队。 |
| 成员 | 启用团队至少一名启用经纪人；每人同一时间只能有一个主团队。 |
| 服务范围 | 至少一个城市范围，区域/商圈必须属于所属城市；成员服务范围必须覆盖其所负责线索。 |
| 分配方式 | 手动分配或按剩余容量轮询；MVP 不支持复杂评分。 |
| 数据权限 | 本团队或含下级团队；作用于主管查看成员线索和跟进记录，不授予跨城市权限。 |
| 排序号/备注 | 排序号为非负整数；备注 500 字内。 |

成员加入时系统校验商户、账号、经纪人状态、服务范围和原主团队；转移后保留原团队关系的起止时间。保存采用版本号校验，冲突时阻止覆盖并提示刷新。

## 9. 核心操作与流程

### 9.1 启用、成员和线索分配

1. 新建团队为草稿；设置城市、范围、主管、成员和分配方式后提交启用。
2. 启用条件：所属城市启用、主管合格、至少一名启用成员、服务范围有效且不超出主管服务范围、层级无循环。
3. 团队候选成员必须是启用、双核验有效、商户启用、未归属其他主团队的经纪人；成员可在启用后因自身状态变化变为不可分配，但仍保留团队历史关系。
4. 团队分配时，仅在本团队内选择经纪人管理 PRD 定义的“线索分配候选人”；按剩余容量轮询时，计算 `日容量 - 当前未完成线索数`，剩余容量为 0 的成员跳过。
5. 线索负责人确定后，写入当时团队 ID 快照；后续成员转队不回写历史跟进记录，实时数据权限按新团队关系生效。

### 9.2 成员转移

将成员转至新主团队时，目标团队必须启用、同城市、服务范围覆盖成员负责的未完成线索，并有合格主管。系统展示成员、其未完成线索和前台顾问关系影响：

1. 目标团队服务范围覆盖全部未完成线索时，成员可带着线索转移，线索团队归属自转移生效时更新；历史快照保留。
2. 存在不覆盖线索时，必须逐条转派给目标团队合格成员或放入公海；不得以成员转队绕过服务范围。
3. 成员的商户归属不因团队转移改变；其前台展示资格仅在新主团队启用且范围仍覆盖关联楼盘或关联房源时维持，否则前台顾问即时隐藏，直至关系重新合格。

### 9.3 停用影响、转移与删除

停用和删除均须先预览成员数、主管、子团队、可分配成员、未完成线索、关联楼盘/关联房源顾问关系和下级团队数据权限影响：

1. 停用团队前必须将全部启用成员转入一个或多个启用目标团队，或先停用成员并按经纪人规则转移其未完成线索；团队主管须同步更换或停用。停用团队不得保留可分配成员。
2. 所有未完成线索必须转移至满足范围和容量的接收经纪人；无候选人时放入公海并记录“团队停用”。已结束线索保持原团队快照。
3. 团队停用立即撤销该团队及其主管基于团队获得的实时数据权限，禁止新线索分配；成员若尚未完成迁移不能提交停用。前台顾问资格按成员新团队重算，不合格者立即隐藏。
4. 有子团队时，必须先将子团队迁移至新上级或停用；不得删除或停用后形成孤儿树节点。
5. 团队存在成员、未完成线索、子团队、数据权限引用、看板统计或审计记录时不得物理删除，只能逻辑删除。逻辑删除前必须完成前四项迁移；删除后不在默认列表和候选项中出现，历史快照、聚合表现和审计保留。

## 10. 状态与业务规则

| 对象 | 枚举与规则 |
|---|---|
| 团队状态 | 草稿、启用、停用、已删除；草稿/停用可启用，已删除不可恢复。 |
| 团队层级 | 同城树状关系；一个团队最多一个上级，不允许循环或跨城市挂载。 |
| 主团队关系 | 经纪人同一时间仅一个；关系具有生效和失效时间。 |
| 数据权限 | 主管按本团队或含下级团队读取实时成员、线索与跟进；转出成员即时失去旧团队实时可见性。 |
| 分配资格 | 团队启用且成员满足经纪人候选条件；团队状态不能覆盖成员核验、商户、范围或容量限制。 |

团队名称、主管姓名和城市不是唯一主键；内部团队 ID 不变。服务范围缩减、主管失效、成员商户/核验失效或上级团队停用时，系统标记受影响对象并阻止新增分配；存量线索按转移规则处置。批量成员操作需展示影响数量、二次确认、逐条结果和失败原因。

## 11. 异常与边界

- 团队树、列表或详情加载失败显示重试并保留当前节点和筛选；无结果显示空状态。
- 无团队数据权限时不可通过下级计数、导出行数或错误提示推断其他团队数据。
- 并发调整成员或主管时以版本号检查，旧版本保存被阻止；转移过程中新增线索时重新预览。
- 目标团队停用、城市不一致、范围不覆盖、容量已满或主管失效时，成员转移或线索分配均失败并返回具体对象原因。
- 团队停用引发的资格重算在 5 分钟内完成，未完成时前台按隐藏顾问处理，后台显示重算中。
- 已停用城市、字典、商户或账号可展示历史数据，不得用于新建、迁移目标或线索分配。

## 12. 数据与指标

数据来自团队主数据及层级、账号/角色、经纪人主团队关系、商户状态、服务范围、楼盘/房源顾问关系、线索/跟进、看板聚合和审计。成员数仅计当前有效关系；可分配成员数计符合经纪人分配候选条件者；未完成线索去重计负责人当前团队为本团队且阶段未结束的线索。

团队聚合表现包括新增线索数、已分配线索数、有效线索数、首次跟进及时率和今日跟进数；计算使用团队生效时的归属快照，避免成员转移回写历史。统计采用北京时间，列表实时查询或缓存不超过 5 分钟，转移/启停用后立即失效。导出继承查询和数据权限，记录导出审计。

## 13. 埋点、通知与审计

埋点包括树节点选择、查询、团队新增/编辑、主管变更、成员加入/移除/转移、范围变更、分配预览/提交、启用、停用、删除预览/确认、详情和看板跳转。主管变更、成员转移、范围缩减和停用影响通知原/新主管、受影响经纪人及城市运营；无法转移而进入公海的线索通知接管主管。

审计字段包括操作人、团队 ID、上级/主管/成员前后快照、数据权限、范围、状态、影响计数、每条线索转移结果、公海原因、时间（秒）和 IP（可选）。历史团队快照和审计记录不可修改或删除。

## 14. 验收标准

| 类别 | 可测试条件 |
|---|---|
| 正常 | 启用团队具备有效城市、合格主管、成员和范围后，可向本团队符合条件的成员分配线索。 |
| 正常 | 成员转入范围覆盖的启用团队时，其实时团队数据权限和线索团队归属自生效时间更新，历史快照保留。 |
| 权限 | 销售主管只能查看本团队及授权下级团队数据，无法修改层级、主管和跨团队数据范围。 |
| 异常 | 目标团队跨城市、停用、范围不覆盖或成员满载时，转移和分配被阻止并给出失败对象原因。 |
| 数据 | 停用团队前未转移成员、子团队或未完成线索时不可提交；停用后无新分配、旧主管失去实时团队数据权限，相关前台顾问按新团队资格隐藏或保留。 |
| 数据 | 有历史关联的团队仅可逻辑删除；删除后不作为候选团队，历史线索、表现和审计仍可追溯。 |

## 15. 依赖与风险

依赖账号/角色、城市区域、商户与经纪人资格、楼盘/房源顾问关系、线索池/公海/跟进、数据看板和操作日志。风险包括历史成员缺少主团队、主管角色与经纪人身份不一致、范围更新引发大批量转移，以及团队停用导致短时前台顾问覆盖下降。上线前须完成团队树和成员关系清洗、权限回算、跨城数据检查及停用/删除演练。
