## 2. 页面概述

**业务背景**：销售过程需要留下可审计的沟通、到访、带看和推荐证据，供负责人执行下一动作并供销售主管检查质量。**页面目标**：提供跨客户的跟进记录检索、详情、受控更正、关联对象追溯及导出。**成功标准**：每条记录的作者和创建时间不可篡改，内容与关联客户/线索、楼盘/房源和阶段变更可以完整还原。

**MVP 范围**：电话、微信、到访、带看和其他渠道的记录；附件、下一跟进、受控更正、筛选、导出和审计。**非目标**：电话外呼、通话录音、IM 同步、自动质检和预约排期。

## 3. 用户与权限

| 角色 | 数据范围与操作权限 |
|---|---|
| 平台管理员 | 全部城市、团队和跟进记录；可查看 README 的 `MVP-R1` 发布常量、审计全部版本，并可在填写原因后执行更正覆盖；本期无 UI 配置入口。 |
| 运营人员 | 授权城市只读脱敏记录；不可新增、审批、更正、分配、认领或变更 MVP-R1 发布常量。 |
| 销售主管 | 所属团队及授权城市；可查看团队记录、审批创建后 24 小时内的更正、执行团队内分配或团队认领并导出授权范围。团队认领须按公海客户规则选择团队内经纪人作为受益归属人；不可变更 MVP-R1 发布常量。 |
| 经纪人 | 仅本人负责线索；可新增记录，并在原记录创建后 24 小时内提交更正申请。 |
| 客服 | 授权城市内本人录入或获授权咨询线索的只读脱敏记录；不可新增销售跟进、审批、更正、分配、认领或变更 MVP-R1 发布常量。 |

客户手机号、附件下载、完整内容和导出列均需字段/导出权限，且不能通过关联跳转绕过。

跟进记录关联 `customer_id` 与 `lead_id`：客户是持久联系人档案，线索是本次需求并承载生命周期；记录不定义客户阶段或第二套线索状态。

## 4. 页面入口与关联

- 菜单入口：客户线索 > 跟进记录；也可从线索池、我的客户、公海客户详情的时间线进入。
- 上游：负责人新增记录、系统生成的分配/转派/认领/回收记录、受控更正申请。
- 下游：客户与当前线索详情、关联楼盘/房源详情、工作台下次跟进提醒、操作日志。
- 前置条件：人工新增必须具有该 `lead_id` 的跟进权限；关联房源推荐时房源须已上架。返回来源页时保留来源筛选条件。

## 5. 页面结构

页面由展开筛选区、记录统计与导出操作栏、按创建时间倒序的列表、详情抽屉和更正申请弹窗组成。列表默认每页 20 条、支持 20/50/100 条；详情依次展示原始内容、更正版本、关联对象、下一动作、系统事件和审计信息。记录作者和首次创建时间固定展示且不可编辑。

## 8. 详情及表单

| 分组 | 字段、校验与行为 |
|---|---|
| 新增跟进 | 渠道必填；内容至少 5 个字；结果必填；下次跟进可选但若填写必须晚于创建时间；作者和创建时间由服务端写入。 |
| 关联对象 | 客户、线索必填；楼盘可选；推荐房源可多选但必须已上架；带看记录至少关联一个楼盘或房源。 |
| 附件 | 可选截图、沟通纪要或现场资料；单个不超过 10MB，保存文件元数据、上传人和时间，下载受权限控制。 |
| 阶段影响 | 记录可作为有效商机、带看、谈判、已转化/已流失的证据；流失必须选择原因，转化/谈判须有说明。 |
| 更正申请 | 原作者仅可在原记录创建后 24 小时内选择更正字段、填写原因和新内容，并提交销售主管审批；平台管理员可超出 24 小时覆盖，但必须填写原因。批准或覆盖后生成新版本，保留原文、作者、首次创建时间、申请人、审批人和时间。 |

不允许直接编辑或删除原记录；更正不会改变原作者、原始创建时间和原内容。保存失败定位字段并保留草稿，版本冲突要求刷新后重试。

## 9. 核心操作与流程

1. **创建**：负责人选择渠道并提交内容、结果、关联对象、附件及下一时间；服务端写入作者、创建时间和版本，并更新线索最后跟进时间、结果摘要及工作台提醒。
2. **带看与推荐**：带看为实际发生记录，须关联楼盘或房源；推荐结果须关联至少一条已上架房源。后续下架不删除历史关联。
3. **阶段变更**：负责人基于记录证据在客户详情或同一提交中申请变更，按第 10 节校验；阶段事件以系统记录写入时间线，不能由人工文本替代。
4. **更正**：原作者在创建后 24 小时内提交申请，记录仍显示原版本；所属团队销售主管审批通过才生成更正版本，拒绝时保留申请和理由。平台管理员可超出窗口覆盖，但必须填写原因并通知原作者和所属销售主管。更正不得隐藏原内容，原作者、原始创建时间和版本链永久保留。若批准或覆盖将记录更正为无效，必须在同一事务发布不可变 `follow_up_invalidated`，重算该 `lead_id` 的 `last_valid_follow_up_at`、`next_follow_up_at` 和阶段证据；若其是当前归属周期首条有效跟进且 SLA 实例仍属该周期，则按原 `due_at` 重开或更新首次跟进待办，已过期立即超时且不得延长。72 小时回收从重算后的最后有效跟进或归属起点计时；仅由该记录支撑的阶段回退到最近有效证据阶段，终态不可自动回退，须进入销售主管纠错流程。
5. **导出**：具有权限的用户导出当前查询集；系统异步生成，继承数据/字段权限，敏感内容和手机号按权限脱敏，并记录导出日志。

## 10. 状态与业务规则

### 10.1 唯一线索生命周期

本模块四个页面使用完全相同的线索生命周期：`新建（new） → 待分配（pending assignment） → 跟进中（following） → 有效商机（valid opportunity） → 带看（viewing） → 谈判（negotiating） → 已转化（converted）/已流失（lost）`。

- 新建为接入校验通过的短暂状态，随后进入待分配；分配给经纪人后进入跟进中。
- 经需求确认可进入有效商机；有实际带看记录可进入带看；有谈判说明可进入谈判。销售主管可补充原因后回退到跟进中或有效商机，并保留阶段历史。
- 已转化和已流失为终态，不能分配、认领或放入公海；已流失必须选择流失原因。已转化只记录线索转化事实，不表示合同、支付或结算。
- 公海是归属位置，不是阶段；新建、待分配、跟进中、有效商机、带看、谈判六个非终态线索可按规则流入或移出公海。

### 10.2 记录一致性

客户由 `customer_id` 标识且不承载销售阶段；线索由 `lead_id` 标识并承载上述阶段。跟进渠道和跟进结果是记录属性，不得被映射为生命周期。系统记录与人工记录均不可物理删除；所有版本、附件、来源、楼盘/房源关联、阶段事件和作者信息均需可追溯。有效人工跟进必须由有权限用户提交，内容不少于 5 个字、结果校验通过，且当前有效版本未撤回；系统记录、草稿、被拒绝或被后续更正为无效的版本不计入 4 小时 SLA 或 72 小时回收计时。

每次分配成功或认领成功均在归属历史中持久化不可变事件 ID、事件类型、负责人和 `ownership_acquired_at`，并在同一事务创建 `first_valid_follow_up_sla_instance`，至少保存 `sla_instance_id`、`lead_id`、负责人、归属事件 ID、起算时间、`due_at=ownership_acquired_at+4` 小时、`mvp_r1_version`、实际 SLA 秒数和关闭状态。预警和逾期均由该实例的 `due_at` 驱动。当前归属期内首条有效人工跟进关闭该实例；重新分配或重新认领必须以本次取得归属时间新建实例，不得使用 `created_at`、首次分配时间或继承旧实例。更正为无效时，同一事务发布 `follow_up_invalidated` 并重算有效跟进、下一待办和阶段证据；首条有效跟进失效且实例仍属同一归属周期时，按原 `due_at` 重开待办，逾期立即超时，不得延长。

线索合并默认保留非终态候选中原始 `created_at` 最早者，相同时取 `lead_id` 字典序最小者；多个非终态线索的需求、阶段或归属冲突时，由销售主管选择目标并填写原因；仅有终态候选时按原始 `created_at`、`lead_id` 顺序确定。来源写入非生命周期字段 `merge_status=已合并` 和 `merged_into_lead_id=目标 lead_id`；目标保留所有跟进版本、来源、归因和历史。来源从新增、分配 SLA、首次有效跟进 SLA 和漏斗排除，指标只计目标一次；仅新增线索、分配 SLA 和漏斗“新建”阶段使用目标及来源最早原始 `created_at` 形成的 `effective_created_at`。

### 10.3 MVP-R1 发布常量与事件快照

| 规则 | README 的 MVP-R1 发布常量 | 执行与事件快照 |
|---|---|---|
| 分配 SLA | 2 小时 | 待分配及成功分配/认领事件持久化 `mvp_r1_version` 和实际秒数。 |
| 首条有效人工跟进 SLA | 最近一次取得归属的分配成功或认领成功时间 + 4 小时 | 每次取得归属时原子固化实例的起算、`due_at`、`mvp_r1_version` 和实际秒数。 |
| 公海认领后归属保护期 | 24 小时 | 认领成功事件固化 `mvp_r1_version`、实际秒数和保护截止时间。 |
| 无有效人工跟进回收 | 连续 72 小时 | 回收判定记录 `mvp_r1_version` 和实际秒数；从重算后的最后有效跟进或归属起点计时。 |
| 每日成功认领上限 | 每名经纪人 20 条/北京时间自然日 | 以受益经纪人、北京时间日界的不可变 `claim_succeeded` 事件 `COUNT` 原子校验，只计成功事件。 |
| 活跃自有线索上限 | 每名经纪人 100 条 | 事务内锁定/串行化受益经纪人，权威 `COUNT(owner_id=该经纪人 AND 非终态 AND merge_status!=已合并)`；不持久化可增减计数。 |
| 跟进更正窗口 | 原记录创建后 24 小时 | 更正事件持久化 `mvp_r1_version` 和实际窗口秒数；窗口内须销售主管审批，平台管理员可在填写原因后覆盖。 |

以上均为 README 定义的 `MVP-R1` 发布常量，本期不提供 UI 配置入口，也不得声称可按人、团队或运行时修改；相关事件必须持久化常量版本及实际值。销售主管可在所属团队执行分配、团队认领、回收与更正审批；团队认领按公海客户规则以受益经纪人为归属、配额和活跃线索校验对象。

## 11. 异常与边界

- 经纪人试图为非本人线索新增记录、关联越权客户或导出他人记录时拒绝并提示无权限。
- 终态线索不可新增常规跟进；销售主管或平台管理员可在审计模式下补充说明，且不得改变终态或伪造原作者/时间。
- 下架、已出租或逻辑删除房源不能用于新推荐/带看；历史记录保留对象 ID 和快照。
- 附件超限、病毒扫描失败、文件类型不允许、内容不足、时间非法或结果与关联对象不匹配时阻止提交并定位原因。
- 查询、导出、上传失败提供重试；导出异步失败可查看失败原因；并发更正以版本号阻止覆盖。

## 12. 数据与指标

数据来自跟进记录、线索、客户、楼盘房源、附件、提醒、操作日志和导出任务；列表实时，指标按日刷新。统计先将 `merged_into_lead_id` 映射到目标，排除 `merge_status=已合并` 的来源线索；目标只计一次，跟进指标读取目标保留的当前有效记录版本。数据层保存归属、认领、SLA、回收和更正事件命中的 `mvp_r1_version` 与实际值；README 的 `MVP-R1` 发布常量为分配 2 小时、首条有效人工跟进 4 小时、认领后保护 24 小时、无有效人工跟进回收 72 小时、每日成功认领 20 条、活跃自有线索 100 条和跟进更正 24 小时，本期无 UI 配置入口。

| 指标 | 口径 |
|---|---|
| 新增跟进数 | 统计期内首次创建的人工跟进 `record_id` 数，不因更正版本重复计数。 |
| 首次有效跟进及时率 | 分子为统计期内 `due_at` 落入周期、在该归属周期 `due_at` 前完成首条有效人工跟进的 `sla_instance_id` 数；分母为同口径已到期或已完成的全部 `sla_instance_id` 数。每次分配、认领、重分配和重认领均独立计数；未到期且未完成单列，不得使用 `created_at`、首次分配时间或按 `lead_id` 去重，明细保留关联 `lead_id`。 |
| 按时跟进率 | 统计期内在下次跟进时间前完成至少一条有效人工跟进的应跟进目标 `lead_id` 数 / 应跟进目标 `lead_id` 总数。 |
| 带看记录数 | 统计期内渠道为带看且成功保存的去重 `record_id` 数。 |
| 推荐房源记录数 | 统计期内结果为已推荐且关联至少一条房源的去重 `record_id` 数。 |

金额、面积、时间和导出口径与线索池一致；导出保留到秒并记录查询条件、字段、数量、操作者和结果。

## 13. 埋点、通知与审计

埋点：`follow_up_created`、`follow_up_corrected_requested`、`follow_up_corrected_approved`、`follow_up_corrected_overridden`、`follow_up_invalidated`、`follow_up_exported`、`viewing_record_created`、`listing_recommended`。进入待分配时通知销售主管，2 小时未分配时发送逾期通知；每次分配或认领成功后展示该 SLA 实例的 `due_at`，到达 `due_at` 未关闭时预警并通知负责人和销售主管。跟进失效导致首条待办重开且已过期时立即发送超时提醒。下次跟进创建后提醒负责人，到期或逾期通知负责人和销售主管；连续 72 小时无有效人工跟进触发回收时通知原负责人和销售主管；认领通知展示 24 小时保护截止。24 小时内提交更正时通知销售主管，审批结果通知原作者；平台管理员覆盖时通知原作者和所属销售主管并展示原因。每日认领 20 条和活跃自有线索 100 条上限在相关提交时明确提示。阶段变更、附件下载、更正、导出均写审计；审计字段包括操作人、`record_id`、`lead_id`、客户 ID、操作类型、前后值、归属事件 ID、SLA 实例 ID、起算/截止时间、原始/更正版本、`mvp_r1_version`、实际常量值、来源页面、IP、时间到秒和结果。

## 14. 验收标准

- **正常**：负责人可为本人线索写入电话、微信、到访、带看或其他记录，关联对象和下次待办正确更新；推荐仅可选已上架房源；首条有效人工跟进默认须在最近一次分配成功或认领成功时间 +4 小时内完成，未关闭实例在该截止时点预警；重分配或重新认领创建新的 SLA 实例。
- **权限**：经纪人仅见本人记录；销售主管仅见所属团队并审批 24 小时窗口内更正，不能变更 MVP-R1 发布常量；平台管理员超窗覆盖必须填写原因；运营人员和客服仅按授权范围只读脱敏数据。
- **异常**：原记录不能直接编辑或删除；更正必须生成可追溯新版本；并发更正使用版本号阻止覆盖；附件失败和越权关联均不能写入错误数据。
- **数据**：记录页面不发明阶段，公海仅表示归属；2 小时分配 SLA、24 小时认领保护、72 小时无有效人工跟进回收、每日认领 20 条及活跃自有线索 100 条均携带 `mvp_r1_version` 与实际值；合并来源写入 `merge_status=已合并` 与 `merged_into_lead_id` 并从新增、SLA、漏斗排除，目标只计一次，作者、首次创建时间、系统事件、关联对象和导出操作全可审计。

## 15. 依赖与风险

依赖线索池、我的客户、公海客户、账号团队权限、README 的 MVP-R1 发布常量、楼盘房源状态、附件存储与扫描、工作台提醒、字典配置、导出任务和操作日志。风险：记录不可变性依赖服务端审计、版本存储和并发校验；不能将跟进结果与线索阶段混用。
