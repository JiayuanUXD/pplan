## 2. 页面概述

**业务背景**：经纪人需要在同一页面理解其负责客户的联系信息、当前需求、关联楼盘/房源、跟进时间线和下一动作。**页面目标**：以客户档案为中心呈现其当前处理线索，支持持续跟进和规范的转化/流失处理。**成功标准**：负责人可在一个详情中完成需求维护、跟进、带看记录、推荐和阶段更新，主管可检查团队进度。

**MVP 范围**：客户档案、当前线索需求、偏好区域/预算、关联楼盘房源、时间线、带看记录、转派、流失和转化。**非目标**：合同、支付、结算、完整预约带看排期、自动推荐。

## 3. 用户与权限

经纪人仅可查看和操作本人负责的客户及其当前线索；销售主管可查看所属团队；管理员按授权城市查看全部；运营/客服只读授权范围且手机号脱敏。负责人本人、主管、管理员可按字段权限查看完整手机号；导出须另有导出权限且继承当前数据范围。

客户档案与线索分离：客户是由 `customer_id` 标识的持久联系人/企业资料；线索是由 `lead_id` 标识的一次咨询或需求，关联客户、来源、归因和生命周期。客户本身不使用销售阶段，本页的阶段字段始终指“当前线索阶段”。

## 4. 页面入口与关联

- 菜单入口：客户线索 > 我的客户；线索被分配或从公海认领后进入。
- 上游：线索池分配、转派和公海认领；客户档案可包含多个历史线索。
- 下游：跟进记录详情、关联楼盘/房源详情、工作台下次跟进待办、公海客户、已转化客户的只读历史。
- 返回列表时保留 URL 中筛选、排序和分页；直接访问客户详情时必须同时校验客户和当前线索的数据权限。

## 5. 页面结构

页面包含固定筛选区、客户列表和客户详情抽屉/详情页。列表默认按下次跟进时间升序、逾期优先；无下次跟进时间的记录按最后跟进时间升序。详情按“客户档案、当前需求、偏好位置与预算、关联楼盘/房源、跟进时间线、带看记录、阶段与归属历史”排序，当前需求和下一动作固定展示。

## 6. 查询与筛选

| 控件 | 规则 |
|---|---|
| 关键词 | 客户姓名、公司、手机号和客户/线索 ID；手机号遵守字段权限。 |
| 当前线索阶段 | 使用第 10 节统一枚举，多选。 |
| 意向城市、区域、商圈 | 城市级联启用的区域/商圈；支持偏好地点多选。 |
| 面积、预算、入驻时间、租期 | 数值区间和日期；面积最小值不得大于最大值，预算保留币种、周期和单位。 |
| 下次跟进、最后跟进、带看时间 | 北京时间范围；支持“今日到期”“已逾期”。 |
| 负责人、团队、来源、是否有关联房源 | 均受数据权限限制；来源包含前台详情咨询、列表快速咨询、委托找房、电话咨询等。 |

重置后回到默认排序；查询状态保留在 URL 和当前会话中。

## 7. 列表或内容展示

| 字段 | 展示规则 |
|---|---|
| 客户 | 姓名、公司、按权限脱敏手机号；客户 ID 不随合并变化。 |
| 当前需求摘要 | 意向城市/区域、面积、预算、入驻时间；无值显示 `-`。 |
| 当前线索阶段 | 仅使用第 10 节阶段；不得显示客户阶段。 |
| 偏好地点 | 区域、商圈、地铁/办公点等由需求字段提供的偏好。 |
| 负责人/团队 | 当前处理线索负责人；历史负责人不覆盖。 |
| 关联楼盘/房源 | 数量及最近关联对象；已下架对象显示历史标识。 |
| 最后跟进/下次动作 | 时间、跟进结果摘要、下次时间和逾期标记。 |
| 操作 | 详情、跟进、编辑需求、推荐、阶段更新、申请转派；按权限显示。 |

列表支持按下次跟进、最后跟进、创建时间排序，空值固定排末；默认每页 20 条，可切换 20/50/100 条。

## 8. 详情及表单

| 分组 | 字段、规则和保存行为 |
|---|---|
| 客户档案 | 姓名、规范化手机号、公司名称；手机号变更需重新查重。客户合并只能由主管/管理员在符合授权的去重流程中执行，保留原始历史。 |
| 当前需求 | 意向城市必填；偏好区域/商圈、面积、预算、工位数、入驻日期、租期、特殊要求；区间校验与线索池一致。保存更新当前 `lead_id`，不改写其他历史线索。 |
| 关联对象 | 可关联楼盘和已上架房源；推荐房源必须已上架。历史推荐即使后续下架仍保留快照和 ID。 |
| 带看记录 | 记录实际带看时间、楼盘/房源、参与人、结果和备注；不是预约排期。带看记录可作为进入“带看”的业务证据。 |
| 下一动作 | 跟进方式、内容、下次跟进时间；内容至少 5 个字。下次时间保存后生成站内待办。 |
| 转化/流失 | 已转化记录时间和说明；已流失必须选择字典中的流失原因。二者均写日志且不可用作合同或结算凭据。 |

保存时使用版本号校验；发生冲突时阻止覆盖，展示最新字段和更新时间。附件遵循单个 10MB 上限、权限和审计规则。

## 9. 核心操作与流程

1. **新增跟进**：负责人选择电话、微信、到访、带看或其他，填写内容、结果、关联对象和下次时间；成功后更新当前线索的最后跟进时间、结果摘要和工作台提醒。
2. **推荐与需求更新**：推荐时至少关联一条已上架房源；需求变化保存到当前线索并记录前后值。系统不自动改变阶段，负责人根据证据选择阶段。
3. **阶段变更**：按第 10 节前进或由主管在有原因时回退；进入带看需存在实际带看记录，进入谈判需有谈判说明，进入已转化或已流失需填写对应信息。
4. **转派/释放**：负责人提交目标经纪人或公海释放原因；主管/管理员校验城市、在职和数据范围后确认。转派和释放均保留全量归属历史并发送站内通知。
5. **客户合并**：规范化手机号命中时仅主管/管理员可手工合并；合并客户档案但不丢失各 `lead_id` 的独立来源、归因、负责人和时间线。

## 10. 状态与业务规则

### 10.1 唯一线索生命周期

本模块四个页面使用完全相同的线索生命周期：`新建（new） → 待分配（pending assignment） → 跟进中（following） → 有效商机（valid opportunity） → 带看（viewing） → 谈判（negotiating） → 已转化（converted）/已流失（lost）`。

- 新建为接入校验通过的短暂状态，随后进入待分配；分配给经纪人后进入跟进中。
- 经需求确认可进入有效商机；有实际带看记录可进入带看；有谈判说明可进入谈判。允许主管补充原因后回退到跟进中或有效商机，历史不可覆盖。
- 已转化和已流失为终态，不能分配、认领或放入公海；已流失必须有流失原因。已转化仅表示线索转化事实，不表示合同、支付或结算完成。
- 公海是归属位置，不是阶段；新建、待分配、跟进中、有效商机、带看、谈判六个非终态线索可按规则进入或移出公海。

### 10.2 客户、线索与数据关系

客户是长期联系人/企业档案，不设销售阶段；线索是客户某次需求及其生命周期实例。一个客户可关联多个线索，列表优先展示当前处理线索；历史和终态线索只读保留。所有楼盘、房源、客户、线索和账号都使用不可变 ID，姓名、手机号和地址不是唯一主键。

## 11. 异常与边界

- 无当前负责人、客户仅有终态历史线索或负责人停用时，详情只读并提示主管转派或认领路径。
- 无权限、直达越权链接、字段越权和导出越权均拒绝访问，不能从关联对象绕过。
- 已下架/已出租房源不可新增推荐或带看关联；已保存的历史记录继续展示对象快照。
- 加载失败可重试；保存失败不丢失草稿；并发编辑提示刷新并对比最新变更。
- 客户手机号重复、城市停用、预算/面积区间非法、附件超限或跟进内容不足时定位字段并阻止提交。

## 12. 数据与指标

实时数据来自客户、线索、跟进、楼盘、房源、归属与操作日志；工作台待办按保存后实时更新，分析报表按日刷新。

| 指标 | 口径 |
|---|---|
| 我的待跟进客户数 | 本人负责且当前线索非终态、下次跟进时间在统计时点前的去重 `customer_id` 数。 |
| 推荐房源率 | 统计期内至少有一次已上架房源推荐的非终态 `lead_id` 数 / 统计期内本人跟进的非终态 `lead_id` 数。 |
| 带看线索数 | 统计期内首次进入带看的去重 `lead_id` 数。 |
| 转化数 | 统计期内首次进入已转化的去重 `lead_id` 数。 |

导出以当前列表为准，继承筛选、排序、数据和字段权限，时间到秒并记录导出审计。

## 13. 埋点、通知与审计

埋点：`customer_detail_viewed`、`lead_requirement_updated`、`follow_up_created`、`listing_recommended`、`viewing_record_created`、`lead_stage_changed`、`lead_transfer_requested`。下次跟进到期提醒负责人，逾期提醒负责人和主管，转派完成通知新旧负责人。审计记录操作者、客户/线索 ID、前后值、时间、IP、来源页面与结果；跟进作者和创建时间不可改写。

## 14. 验收标准

- **正常**：经纪人可维护本人客户的当前需求、推荐已上架房源、提交带看和跟进记录，并生成下次待办。
- **权限**：经纪人不能查看他人客户，主管仅看团队；未授权用户不能查看完整手机号或导出敏感列。
- **异常**：下架房源不能新增推荐；并发保存和手机号重复均阻止覆盖并保留用户输入。
- **数据**：客户档案不出现销售阶段，所有阶段来自当前 `lead_id` 且仅为第 10 节枚举；客户合并后来源、归因和历史完整可查。

## 15. 依赖与风险

依赖线索池、跟进记录、楼盘/房源上下架状态、经纪人服务城市与团队、线索来源/流失原因字典、工作台提醒、操作日志和导出服务。风险：多线索客户的“当前处理线索”选择规则需以最近非终态且最近更新的 `lead_id` 固化；不得把客户档案状态误作线索生命周期。
