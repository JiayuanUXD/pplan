## 2. 页面概述

**业务背景**：无人承接、主动释放或被回收的可继续处理需求需要公开、可控地重新分配，不能丢失客户与历史。**页面目标**：管理在公海位置的线索，提供公平认领、销售主管分配、回收与冲突处理。**成功标准**：每条公海线索可解释入池原因、认领后保护期、历史负责人和认领结果，且历史全量保留。

**MVP 范围**：入池、认领后保护期、认领资格和限额、销售主管分配、批量分配、超时回收、冲突处理和历史查看。**非目标**：自动抢单、积分竞价、复杂激励和交易结算。

## 3. 用户与权限

| 角色 | 数据范围与操作 |
|---|---|
| 平台管理员 | 全部城市、团队和公海线索；可查看 README 的 `MVP-R1` 发布常量、处理跨团队冲突和审计全部操作；本期无 UI 配置入口。 |
| 运营人员 | 授权城市只读脱敏数据，不可认领、分配、审批、回收或变更 MVP-R1 发布常量。 |
| 销售主管 | 所属团队及授权城市；可查看、团队内分配或发起团队认领、回收、处理冲突和查看团队完整手机号。团队认领必须选择一名团队内经纪人作为受益归属人；不可变更 MVP-R1 发布常量。 |
| 经纪人 | 服务城市内且符合认领资格的公海线索；可查看脱敏信息、认领本人成功后的完整信息。 |
| 客服 | 授权城市内本人录入或获授权咨询线索的只读脱敏数据，不可认领、分配、审批、回收或变更 MVP-R1 发布常量。 |

文件名中的“客户”是业务导航名称；公海的可流转对象始终是附着于客户的 `lead_id`。客户档案保持存在，客户本身不进入公海、不使用销售阶段。

## 4. 页面入口与关联

- 菜单入口：客户线索 > 公海客户；也可由线索池释放、销售主管回收、导入待分配和工作台超时提醒进入。
- 上游：非终态线索的主动释放、满足规则的回收、未分配接入；入池时保留客户、需求、来源、楼盘/房源归因、原负责人和全部历史。
- 下游：认领或销售主管分配后进入我的客户；历史记录进入跟进记录；查看时可跳转客户、楼盘和房源详情。
- 直达链接、认领、分配和导出均先校验城市、团队、服务范围、在职状态和数据权限。

## 5. 页面结构

页面从上至下为：统计提示（可认领数、今日剩余额度、活跃自有线索剩余额度）、筛选区、按入池时间倒序的列表、批量分配工具栏、详情抽屉和认领/冲突处理弹窗。列表默认 20 条，支持 20/50/100 条；详情固定显示入池原因、认领后 24 小时保护截止时间、当前线索阶段和完整归属历史。

详情只读展示客户档案、当前线索需求、来源参数、关联楼盘/房源、入池与归属历史、跟进和阶段历史。手机号、附件和关联对象按字段权限展示；已下架对象保留历史快照。

|---|---|
| 入池/释放 | 入池原因必填；主动释放补充说明；仅第 10 节六个非终态可提交。 |
| 个人认领 | 受益归属人为操作人本人；以北京时间自然日内该受益人的不可变 `claim_succeeded` 事件 `COUNT` 校验少于 20 条。事务内锁定/串行化受益经纪人后权威查询 `COUNT(owner_id=该经纪人 AND 非终态 AND merge_status!=已合并)`，认领后不得超过 100 条。 |
| 销售主管团队认领 | 必须选择一名所属团队内、在职、服务意向城市且有认领权限的经纪人作为受益归属人；不得将销售主管本人、空值或团队外人员作为受益人。提交时以该受益经纪人的不可变成功认领事件 `COUNT` 和上述权威容量查询原子校验。 |
| 销售主管分配 | 选择在职、服务意向城市且通过事务内权威容量查询的经纪人；批量时先逐条预校验。 |

表单提交采用版本号校验，错误定位到字段；所有危险动作二次确认。

## 9. 核心操作与流程

1. **入池触发**：待分配线索超过 2 小时分配 SLA、负责人主动释放、连续 72 小时无有效人工跟进触发回收、或导入后暂未分配时，可进入公海。入池不改变线索阶段，只清空当前负责人并记录原负责人、原因和时间。
2. **认领后保护期与 SLA**：认领成功后从 `claimed_at` 起提供 24 小时归属保护；保护期内不得因 72 小时无有效人工跟进规则回收。认领成功的同一原子事务持久化 `claim_succeeded` 归属事件和 `ownership_acquired_at=claimed_at`，并新建首条有效人工跟进 SLA 实例，`due_at=ownership_acquired_at+4` 小时；实例同时持久化 `runtime_policy_version=MVP-R1` 与实际秒数。预警和逾期均由该实例的截止时间驱动。当前归属期的首条有效人工跟进关闭实例；重新认领或重新分配均按本次取得归属时间另建实例，绝不使用 `created_at` 或继承旧实例。保护截止时间在认领原子事务中固化，README 的发布常量不改写既有事件实际值。
3. **认领资格、受益归属与限额**：个人认领的操作人和受益归属人为同一经纪人。销售主管团队认领必须选择一名所属团队内、在职、服务意向城市且有认领权限的经纪人作为受益归属人，销售主管仅为操作人；不得省略受益人或将团队外人员作为受益人。无论个人还是团队认领，均以受益经纪人为准，以北京时间自然日内不可变 `claim_succeeded` 成功事件的 `COUNT` 校验少于 20 条；不维护可变每日配额计数。
4. **冲突处理与原子提交**：认领请求使用服务端原子锁、版本号和条件更新一次性校验公海位置、当前负责人、`merge_status`、受益经纪人的在职/团队/城市/权限及成功认领事件 `COUNT`。同一事务内锁定/串行化受益经纪人，权威查询 `COUNT(owner_id=该经纪人 AND 非终态 AND merge_status!=已合并)`；仅当认领后的查询结果不超过 100 条才可成功。成功时将负责人设为受益经纪人、写入不可变 `claim_succeeded` 归属事件/`ownership_acquired_at`/保护截止/SLA 实例，并分别记录操作人和受益归属人；不得写入活跃自有线索或配额的可增减持久化计数，转交、终态、入公海和合并自然改变查询集合。任一校验或写入失败均整体回滚。首个成功请求获得负责人，其他请求返回“已被认领”及可见的必要信息。权限或数据归属争议由销售主管在所属团队处理，跨团队争议由平台管理员处理并记录理由。
5. **回收与跟进纠错**：连续 72 小时无有效人工跟进时系统触发回收，由销售主管在所属团队执行；计时起点为重算后的最近有效人工跟进时间，尚无有效跟进时取归属起点。跟进更正为无效时，必须在同一事务发布 `follow_up_invalidated`，重算 `last_valid_follow_up_at`、`next_follow_up_at` 和阶段证据；若失效的是当前归属周期首条有效跟进且 SLA 实例仍属该周期，则按原 `due_at` 重开/更新待办，已过期立即超时且不得延长。阶段仅由该记录支撑时回退到最近有效证据阶段，终态不可自动回退，须走销售主管纠错流程。回收后写入系统记录和站内通知，且保留全部跟进、来源和归因。

批量分配显示影响数量、每条成功/失败与原因；认领、分配、回收、MVP-R1 发布常量命中和冲突裁决均写入审计日志。

## 10. 状态与业务规则

### 10.1 唯一线索生命周期

本模块四个页面使用完全相同的线索生命周期：`新建（new） → 待分配（pending assignment） → 跟进中（following） → 有效商机（valid opportunity） → 带看（viewing） → 谈判（negotiating） → 已转化（converted）/已流失（lost）`。

- 新建为接入校验通过的短暂状态，随后进入待分配；分配给经纪人后进入跟进中。
- 经需求确认可进入有效商机；有实际带看记录可进入带看；有谈判说明可进入谈判。销售主管可补充原因后回退到跟进中或有效商机，且必须保留流转历史。
- 已转化和已流失为终态，不能分配、认领、回收或放入公海；已流失必须选择流失原因。已转化只记录转化事实，不扩展合同、支付或结算。
- 公海是归属位置，不是阶段；新建、待分配、跟进中、有效商机、带看、谈判六个非终态线索可按入池、保护、认领和分配规则流入或流出公海。

### 10.2 归属与历史

客户是由 `customer_id` 标识的持久联系人/企业档案；线索是由 `lead_id` 标识的一次需求并承载阶段和公海位置。每次入池、认领、分配、释放和回收新增一条不可变归属历史，不删除原负责人、跟进、来源、附件、楼盘/房源关联或日志。认领归属历史至少持久化事件 ID、个人/团队认领类型、操作人、受益归属人、`ownership_acquired_at`、保护截止、受益人的认领配额/活跃线索校验结果和 SLA 实例 ID；个人认领的操作人与受益归属人相同。规范化手机号去重和手工合并遵循线索池规则，合并不得使历史不可追溯。

线索合并默认保留非终态候选中原始 `created_at` 最早者，相同时取 `lead_id` 字典序最小者；多个非终态线索的需求、阶段或归属冲突时，由销售主管选择目标并填写原因；仅有终态候选时按原始 `created_at`、`lead_id` 顺序确定。来源写入非生命周期字段 `merge_status=已合并` 和 `merged_into_lead_id=目标 lead_id`，不得认领、分配或回收；目标保留全部来源、归因和历史。来源线索从新增、分配 SLA、首次有效跟进 SLA 和漏斗排除，指标只计目标一次；仅新增线索、分配 SLA 和漏斗“新建”阶段使用最早原始 `created_at` 形成的 `effective_created_at`。

### 10.3 MVP-R1 发布常量与事件快照

| 规则 | README 的 MVP-R1 发布常量 | 执行与事件快照 |
|---|---|---|
| 分配 SLA | 2 小时 | 待分配及成功分配/认领事件持久化 `runtime_policy_version=MVP-R1` 和实际秒数。 |
| 首条有效人工跟进 SLA | 最近一次取得归属的分配成功或认领成功时间 + 4 小时 | 每次取得归属时原子固化实例的起算、`due_at`、`runtime_policy_version=MVP-R1` 和实际秒数。 |
| 公海认领后归属保护期 | 24 小时 | 认领成功事件固化 `runtime_policy_version=MVP-R1`、实际秒数和保护截止时间。 |
| 无有效人工跟进回收 | 连续 72 小时 | 回收判定记录 `runtime_policy_version=MVP-R1` 和实际秒数；从重算后的最后有效跟进或归属起点计时。 |
| 每日成功认领上限 | 每名经纪人 20 条/北京时间自然日 | 以受益经纪人、北京时间日界的不可变 `claim_succeeded` 事件 `COUNT` 原子校验，只计成功事件。 |
| 活跃自有线索上限 | 每名经纪人 100 条 | 事务内锁定/串行化受益经纪人，权威 `COUNT(owner_id=该经纪人 AND 非终态 AND merge_status!=已合并)`；不持久化可增减计数。 |
| 跟进更正窗口 | 原记录创建后 24 小时 | 更正事件持久化 `runtime_policy_version=MVP-R1` 和实际窗口秒数；窗口内须销售主管审批，平台管理员可在填写原因后覆盖。 |

以上均为 README 定义的 `MVP-R1` 发布常量，本期不提供 UI 配置入口，也不得声称可按人、团队或运行时修改；相关事件必须持久化常量版本及实际值。销售主管可在所属团队执行分配、团队认领、回收与更正审批；团队认领以受益经纪人为归属、配额和活跃线索校验对象。

## 11. 异常与边界

- 终态、无效接入、被逻辑删除、`merge_status=已合并` 或已被认领的线索不可进入或继续显示为可认领；只读历史可在权限范围内查看。
- 认领并发、页面过期或归属刚变化时，服务端以提交时受益经纪人的资格、版本号和原子锁结果为准，前端刷新列表。
- 团队认领未选择受益经纪人、受益经纪人不在所属团队/已停用/不服务意向城市/无认领权限，或其每日成功认领已达 20 条、认领后活跃自有线索将超过 100 条时，拒绝认领并说明原因；不得改以销售主管的配额或活跃线索数通过校验。
- 加载、查询和导出失败可重试；批量操作部分失败不得回滚已成功的逐条处理，必须返回清单。
- 下架房源、停用城市和历史负责人离职不删除历史归因；不得将其作为新的分配候选。

## 12. 数据与指标

数据来自线索、客户、归属历史、首次有效跟进 SLA 实例、跟进、人员组织、楼盘房源和操作日志；列表实时，统计按日刷新。统计先将 `merged_into_lead_id` 映射到目标，排除 `merge_status=已合并` 的来源并只计目标一次。每日成功认领数按受益经纪人的不可变 `claim_succeeded` 事件 `COUNT` 汇总，操作人不替代受益人；活跃自有线索不持久化计数，实时权威查询 `owner_id` 为受益经纪人、非终态且非合并来源的 Lead 集合。数据层保存归属、认领、SLA、回收和更正事件命中的 `runtime_policy_version=MVP-R1` 与实际值；README 的 `MVP-R1` 发布常量为分配 2 小时、首条有效人工跟进 4 小时、认领后保护 24 小时、无有效人工跟进回收 72 小时、每日成功认领 20 条、活跃自有线索 100 条和跟进更正 24 小时，本期无 UI 配置入口。

| 指标 | 口径 |
|---|---|
| 当前公海线索数 | 统计时点公海位置且当前阶段为非终态的去重 `lead_id` 数。 |
| 公海认领成功率 | 统计期内成功认领数 / 发起且资格校验通过的认领请求数。 |
| 首次有效跟进及时率 | 分母为 `due_at` 落入统计期且已到期或已完成的全部首次有效跟进 `sla_instance_id`，包括分配和公海认领取得归属后创建、重分配或重认领后新建的实例；分子为对应实例在 `due_at` 前完成首条有效人工跟进的数量。按 SLA 实例计数，不以 Lead 当前归属、创建时间或操作人替代；未到期且未完成单列，明细保留关联 `lead_id`，分母为 0 显示 `--`。 |
| 回收线索数 | 统计期内因连续 72 小时无有效人工跟进或销售主管回收进入公海的目标 `lead_id` 数。 |
| 平均公海停留时长 | 统计期内从入池到成功认领/分配的时长总和 / 已流出公海的 `lead_id` 数。 |

时间使用北京时间，金额和面积口径与线索池一致。导出继承当前条件、数据/字段权限并记录审计，不输出超出权限范围的公海记录。

## 13. 埋点、通知与审计

埋点：`lead_entered_public_pool`、`public_pool_claim_attempted`、`public_pool_claimed`、`public_pool_assigned`、`lead_recovered`、`follow_up_invalidated`、`public_pool_conflict_resolved`。`public_pool_claim_attempted` 和 `public_pool_claimed` 均记录个人/团队类型、操作人和受益归属人。通知：进入待分配时通知销售主管，2 小时未分配而进入公海时发送逾期与入池通知；成功认领通知操作人、受益归属人和所属销售主管，并展示该 SLA 实例的 `due_at` 与 24 小时保护截止；到达 `due_at` 未关闭时预警并通知受益归属人和所属销售主管；失效跟进导致首条待办重开且已过期时立即通知超时；连续 72 小时无有效人工跟进触发回收时通知原负责人和销售主管。达到每日认领 20 条或活跃自有线索 100 条时向提交方返回受益经纪人对应的明确拒绝原因。24 小时更正申请和审批通知遵循跟进记录规则。审计记录操作人、受益归属人、`lead_id`、客户 ID、前后负责人、公海原因、归属事件 ID、SLA 实例 ID、起算/截止时间、`runtime_policy_version=MVP-R1`、实际常量值、受益人的资格/限额结果、时间到秒、IP 和处理结果。

## 14. 验收标准

- **正常**：符合条件的非终态线索可入池；个人认领以操作人本人为受益归属人。销售主管团队认领必须选择团队内合格经纪人为受益归属人，成功后该经纪人进入我的客户、写入一条不可变成功认领事件、获得 24 小时归属保护，并以本次取得归属时间生成首条有效人工跟进 SLA 实例（截止及未关闭预警均为 +4 小时）；完整历史保留。
- **权限**：非服务城市、无认领权限或停用经纪人不能认领；受益经纪人的不可变成功认领事件 `COUNT` 已达 20 或认领后权威容量查询超过 100 条时，个人和团队认领均必须拒绝；销售主管仅在所属团队执行团队认领且不能变更或绕过 MVP-R1 发布常量；无字段权限者仅见脱敏数据。
- **异常**：并发认领仅一个请求成功；团队认领缺少受益经纪人、受益人团队/城市/权限不符、成功认领事件 `COUNT` 已达 20 或权威容量查询显示认领后超过 100 条时整体失败且不改变负责人；不得写入配额或活跃线索计数。批量分配显示逐条失败原因；已转化和已流失不可入池或认领。
- **数据**：公海不作为线索阶段；团队认领的操作人、受益归属人、成功认领事件 `COUNT`、权威容量查询、归属事件和 SLA 实例均可追溯，且不维护配额或活跃线索可增减计数；2 小时分配 SLA、按最近一次取得归属时间 +4 小时的首条有效人工跟进 SLA、24 小时认领保护、72 小时无有效人工跟进回收和 24 小时更正窗口均携带 `runtime_policy_version=MVP-R1` 与实际值；合并来源写入 `merge_status=已合并` 与 `merged_into_lead_id` 并从新增、SLA、漏斗排除，目标只计一次且入池、认领、回收、来源和跟进历史均可追溯。
