## 2. 页面概述

我的待办聚合当前登录人需要处理的审核、举报、线索、客户跟进、商户资质和房源维护任务，减少跨菜单查找成本。成功标准是用户能够识别优先事项、在一次跳转内到达可处理页面，并在处理完成后获得明确结果。MVP 不提供通用流程编排器、跨组织协同工单和自定义待办类型。

## 3. 用户与权限

| 角色 | 可见待办 | 可执行操作 |
| --- | --- | --- |
| 平台管理员 | 操作权限覆盖范围内的全平台待办 | 查看、处理、转派、催办；待分配线索须具备线索分配操作权限；商户资质和经纪人身份/任职核验仍须具备对应核验操作权限 |
| 运营人员 | 授权城市的楼盘、房源、举报、线索及商户资料待办；待分配线索仅在另行获得该任务查看授权时可见 | 按来源页面操作权限查看、处理、转派、催办；待分配线索仅可查看，不得分配、认领、代认领或转派；不得执行经纪人身份或任职核验 |
| 销售主管 | 具备线索分配操作权限时，可见所属团队的待分配线索、客户跟进和经纪人停用前线索转移待办 | 查看、分配、转派、催办；经纪人核验信息仅可在数据范围内查看，不得作出核验结论 |
| 经纪人 | 分配给本人的线索和客户跟进待办 | 查看、处理本人任务；其他来源仍按来源页面操作权限判断 |
| 客服 | 授权城市内由本人录入或待补充的咨询线索 | 查看、补充资料、提交处理 |

页面权限只决定能否进入；每条待办还要同时校验城市、商户、团队、本人数据范围。手机号等敏感信息沿用来源页面脱敏规则，待办列表不额外暴露完整信息。

## 4. 页面入口与关联

- 入口：左侧导航“工作台 / 我的待办”；顶部待办角标可携带 `status=pending` 进入。
- 上游来源：房源审核、房源举报、线索池、我的客户、商户管理、经纪人管理及系统生成的维护提醒。
- 下游：点击标题或“去处理”进入具体业务详情，并携带 `from=todo&todoId`；完成后返回时保留筛选、页码和滚动位置。
- 前置条件：登录有效且具有至少一个待办来源页面的查看权限；无任何来源权限时展示无权限状态而不是空列表。

## 5. 页面结构

从上到下为：页面标题与最后刷新时间、汇总卡、筛选区、批量操作栏、待办列表、分页。汇总卡固定为“全部、今日到期、已逾期、高优先级”四项；列表每页默认 20 条。筛选区默认展开一行，高级条件可收起；刷新不改变布局尺寸。

## 6. 查询与筛选

| 条件 | 控件与规则 |
| --- | --- |
| 关键词 | 模糊匹配任务编号、业务对象名称、客户脱敏手机号后四位 |
| 待办类型 | 多选；选项来自系统待办类型配置，只显示本人有来源权限的类型 |
| 状态 | 待处理、处理中、已完成、已取消；默认待处理+处理中 |
| 优先级 | 高、中、低，多选 |
| 截止时间 | 今日、3 日内、7 日内、自定义；按北京时间闭区间 |
| 是否逾期 | 全部、已逾期、未逾期 |
| 来源模块 | 房源、举报、线索、客户、商户、经纪人 |

点击查询后将非敏感条件写入 URL；清空恢复默认状态。日期起始不得晚于结束，跨度最多 180 天。切换汇总卡会覆盖状态或截止时间条件，并将页码重置为 1。

### 7.2 列表字段

| 字段 | 展示规则 |
| --- | --- |
| 任务编号 | 全局唯一，支持复制 |
| 待办类型/标题 | 类型图标+可读标题；点击进入任务注册表规定的来源详情 |
| 业务对象 | 对象类型、名称、不可变内部 ID；对象失效显示“对象已失效” |
| 来源/创建时间 | 明示权威来源模块和来源 ID；北京时间 `YYYY-MM-DD HH:mm` |
| 优先级 | 高红、中橙、低灰；颜色同时配文字 |
| 负责人 | 姓名+团队；无负责人显示“待领取” |
| 截止时间 | 到分钟；已逾期显示逾期时长 |
| 状态 | 待处理、处理中、已完成、已取消 |
| 未读状态 | 当前用户首次看到新建或转派任务前为未读；打开详情即置已读，不改变业务状态 |
| 操作 | 去处理、领取、转派、催办；按权限与状态显示 |

默认排序为：已逾期优先，其次优先级高到低，再按截止时间升序、创建时间升序。支持按截止时间和创建时间切换排序。

## 8. 详情及表单

点击任务编号打开只读抽屉，包含任务信息、权威来源和来源 ID、来源版本、触发规则、SLA、负责人变化、处理记录、未读状态和通知记录。转派弹窗必填新负责人和转派原因（2–200 字），候选人仅包含同数据范围且具备来源页面操作权限的启用账号。领取操作需二次确认；催办弹窗可补充 200 字以内说明，10 分钟内同任务不可重复催办。

## 9. 核心操作与流程

1. 来源业务创建待办，系统计算负责人、优先级与截止时间。
2. 无负责人的任务由有权限用户领取，状态从待处理变为处理中。
3. “去处理”跳转来源页面；来源动作成功后通过业务事件自动完成待办，不允许仅在待办页手工完成。
4. 转派成功后原负责人失去处理入口，新负责人收到站内通知；完整转派链写入日志。
5. 来源对象撤回、删除或进入不再需要处理的终态时，待办自动取消并记录原因。

批量领取和批量转派上限 50 条，仅允许同类型、同城市且状态可操作的任务；结果按成功与失败逐条返回。

## 10. 状态与业务规则

状态流转：待处理 → 处理中 → 已完成；待处理/处理中可因来源失效进入已取消。已完成和已取消为终态。举报进入等待审核时，旧处理待办以 `已提交复核` 和关联 `review_id` 进入已完成；审核驳回或退回后的新处理阶段只能按新的 `processing_stage_version` 生成任务，审核通过的恢复或关闭不得重开任何旧任务。房源信息时效维护的 `source_version_superseded` 属于来源版本失效，旧版本任务只能先进入已取消，随后才允许创建新版本任务。逾期是计算属性：当前时间晚于截止时间且状态非终态。截止时间由任务注册表来源规则生成，人工转派不得自动延期。高优先级不等同于逾期，两者可同时存在；未读状态是用户通知属性，不参与业务状态流转。

## 11. 异常与边界

- 加载中使用固定高度骨架；无数据说明当前筛选下无待办并提供清空筛选。
- 无页面权限展示申请权限提示，不返回任何数量。
- 来源对象失效时禁止处理，允许查看失效原因并刷新任务状态。
- 两人同时领取时，以服务端首次成功为准，后者提示负责人已变化并刷新该行。
- 转派对象在提交前被停用时，提交失败并重新加载候选人。
- 接口失败保留筛选和已选项，提供重试；不得以本地状态假定处理成功。
- 举报复核提交、审核驳回和审核退回事件均使用来源事件 ID 幂等处理：重复提交不得重复完成旧任务，重复回传不得为同一 `report_id+processing_stage_version` 创建多条任务；关联 `review_id` 缺失或不一致时拒绝更新并告警。

## 12. 数据与指标

数据来自统一待办表，来源业务通过幂等事件写入。汇总卡按当前用户数据权限统计，去重键为待办内部 ID。今日到期指截止时间位于北京时间当日 00:00:00–23:59:59；已逾期不包含终态；高优先级包含所有非终态高优先级任务。列表实时查询，汇总卡缓存不超过 60 秒，手动刷新绕过缓存。导出不属于本页 MVP。

## 13. 埋点、通知与审计

埋点：页面访问、汇总卡切换、筛选查询、去处理、领取、转派、催办、操作失败。领取、转派、取消和完成记录操作人、前后负责人、来源对象、请求 ID、北京时间和结果；举报因提交复核完成时额外记录 `report_id`、`review_id`、`processing_stage_version` 和原因 `已提交复核`，审核驳回/退回重开时记录来源事件 ID 和新阶段版本。新分配、被转派、截止前 24 小时、逾期时发送站内通知；相同任务同类提醒每日最多一次。

## 14. 验收标准

- 正常：对任一来源任务完成来源页规定的处理动作后，在 60 秒内复查待办为已完成；列表默认排序严格遵循逾期、优先级、截止时间、创建时间顺序。
- 权限：为运营人员配置待分配线索的任务查看授权但不授予线索分配操作权限。该用户可查看授权数据范围内任务和详情，列表、详情与接口均不得提供或接受分配、认领、代认领、转派操作；直接调用上述操作返回无权限。
- 权限：以具备线索分配操作权限的销售主管或平台管理员完成一次 Lead 分配，并以具备公海认领权限的用户完成一次 Lead 认领；两次分别记录成功事件时刻 `T`，均断言 `first_valid_follow_up_due_at=T+4小时（Asia/Shanghai）`。在该时刻前不生成首次跟进超时任务，达到该时刻且首条有效跟进仍为空时，由 15 分钟扫描生成一条任务；不得以 `created_at` 或 `effective_created_at` 替代 `T`。
- 状态：对同一举报依次构造 `有效已关闭`、`无效已关闭`、`已合并`、`已撤回` 四个终态，分别复查任务状态为已完成、已完成、已完成、已取消；举报来源记录删除时也必须转为已取消并记录原因。
- 状态：举报进入 `等待审核` 时，当前处理待办在 60 秒内以原因 `已提交复核`、关联 `review_id` 转为已完成；对同一审核驳回或退回事件重复投递，举报回到核实中/等待资料后仅存在一条 `report_id+processing_stage_version` 新待办。审核通过后的恢复或关闭不得遗留或重开旧处理待办。
- 数据：设扫描日 `as_of_date=2026-07-17`（`Asia/Shanghai`）。已上架 Listing 仅在满足 `last_effective_maintained_at IS NULL OR DATE(last_effective_maintained_at AT TIME ZONE Asia/Shanghai) <= as_of_date-30` 时命中：`last_effective_maintained_at=NULL` 与 `2026-06-17 23:59:59+08:00` 必须命中，`2026-06-18 00:00:00+08:00` 不得命中；仅变更 `updated_at` 不得改变结果。
- 数据：对同一仍满足 `last_effective_maintained_at IS NULL OR DATE(last_effective_maintained_at AT TIME ZONE Asia/Shanghai) <= as_of_date-30`（包括空值）的 Listing、同一 `listing.version` 连续执行两次每日扫描，第二次使用新的 `as_of_date`；断言仅保留一条非终态 `listing_id+listing_data_quality_stale+source_version` 任务，任务 ID 和创建时间不变，`last_evaluated_at`、`as_of_date`、`stale_before_date` 更新，且不产生按日期累积的任务。
- 数据：将仍满足陈旧谓词的 Listing 的 `source_version` 变更后执行扫描；旧版本非终态任务先以 `source_version_superseded` 转为已取消，再创建当前版本的一条非终态任务。将 Listing 的 `last_effective_maintained_at` 更新至使其不再满足 `last_effective_maintained_at IS NULL OR DATE(last_effective_maintained_at AT TIME ZONE Asia/Shanghai) <= as_of_date-30`，或改为已下架/已出租时，任务必须转为已完成；删除来源记录时必须转为已取消。
- 异常：并发领取只允许一人成功，失败方得到明确反馈且不覆盖负责人；来源对象失效、接口超时和部分批量失败均可恢复，不产生重复任务。
- 数据：用户看不到权限范围外的任务、汇总数量和敏感字段，直接访问待办 ID 返回无权限；转派候选人不包含无来源操作权限、跨数据范围或停用账号；汇总卡、筛选结果与来源页面相同条件下的数据一致，时间均按 `Asia/Shanghai` 计算。

## 15. 依赖与风险

依赖各业务模块提供幂等事件、来源详情深链、负责人和终态定义；依赖账号权限、团队和站内通知服务。主要风险是来源模块状态口径不一致导致任务无法自动闭环，因此每类待办上线前必须登记唯一来源事件、完成事件、取消事件与 SLA。MVP 不支持用户自定义待办规则。
