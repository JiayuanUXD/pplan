# 我的待办页面 PRD

## 1. 文档信息

| 项目 | 内容 |
| --- | --- |
| 文档编号 | BO-MVP-WB-001 |
| 版本 | V1.0 |
| 状态 | 待评审 |
| 所属模块 | 工作台 / 我的待办 |
| 适用端 | PC 管理后台 |
| 时间口径 | 北京时间（UTC+8） |

## 2. 页面概述

我的待办聚合当前登录人需要处理的审核、举报、线索、客户跟进、商户资质和房源维护任务，减少跨菜单查找成本。成功标准是用户能够识别优先事项、在一次跳转内到达可处理页面，并在处理完成后获得明确结果。MVP 不提供通用流程编排器、跨组织协同工单和自定义待办类型。

## 3. 用户与权限

| 角色 | 可见待办 | 可执行操作 |
| --- | --- | --- |
| 平台管理员 | 全平台待办 | 查看、转派、催办；业务操作仍受具体页面权限约束 |
| 运营人员 | 授权城市的楼盘、房源、举报、线索与商户待办 | 按操作权限查看、处理、转派、催办 |
| 销售主管 | 所属团队的线索分配、客户跟进和人员转移待办 | 查看、分配、转派、催办 |
| 经纪人 | 分配给本人的线索、跟进和房源维护待办 | 查看、处理本人任务 |
| 客服 | 授权城市内由本人录入或待补充的咨询线索 | 查看、补充资料、提交处理 |

页面权限只决定能否进入；每条待办还要同时校验城市、商户、团队、本人数据范围。手机号等敏感信息沿用来源页面脱敏规则，待办列表不额外暴露完整信息。

## 4. 页面入口与关联

- 入口：左侧导航“工作台 / 我的待办”；顶部待办角标可携带 `status=pending` 进入。
- 上游来源：房源审核、房源举报、线索池、我的客户、商户管理、经纪人管理及系统生成的维护提醒。
- 下游：点击标题或“去处理”进入具体业务详情，并携带 `from=todo&todoId`；完成后返回时保留筛选、页码和滚动位置。
- 前置条件：登录有效且具有至少一个待办来源页面的查看权限；无任何来源权限时展示无权限状态而不是空列表。

## 5. 页面结构

从上到下为：页面标题与最后刷新时间、汇总卡、筛选区、批量操作栏、待办列表、分页。汇总卡固定为“全部、今日到期、已逾期、高优先级”四项；列表每页默认 20 条。筛选区默认展开一行，高级条件可收起；刷新不改变布局尺寸。

## 6. 查询与筛选

| 条件 | 控件与规则 |
| --- | --- |
| 关键词 | 模糊匹配任务编号、业务对象名称、客户脱敏手机号后四位 |
| 待办类型 | 多选；选项来自系统待办类型配置，只显示本人有来源权限的类型 |
| 状态 | 待处理、处理中、已完成、已取消；默认待处理+处理中 |
| 优先级 | 高、中、低，多选 |
| 截止时间 | 今日、3 日内、7 日内、自定义；按北京时间闭区间 |
| 是否逾期 | 全部、已逾期、未逾期 |
| 来源模块 | 房源、举报、线索、客户、商户、经纪人 |

点击查询后将非敏感条件写入 URL；清空恢复默认状态。日期起始不得晚于结束，跨度最多 180 天。切换汇总卡会覆盖状态或截止时间条件，并将页码重置为 1。

## 7. 列表或内容展示

### 7.1 任务注册表

下表是待办创建、时效、去重与跳转的唯一产品口径。待办只保存来源类型、不可变来源 ID、来源版本和显示快照；审核、举报、线索、资质和发布状态仍以列出的来源页面为权威。

| 任务类型 | 权威来源与触发条件 | 角色可见性 | 数据范围 | 刷新或 SLA | 去重规则 | 下钻目的地及完成条件 |
| --- | --- | --- | --- | --- | --- | --- |
| 待审核房源 | `03_房源管理/02_房源审核_PRD.md`；审核单状态为待审核 | 具有房源审核权限的运营人员；平台管理员可按权限处理或只读查看 | 审核单所属城市和审核业务线 | 状态事件后 60 秒内刷新；提交后 24 小时 SLA，逾期即标红 | `review_id + 审核轮次` | 房源审核详情；通过、驳回或退回补充后自动完成 |
| 房源举报待处理 | `03_房源管理/03_房源举报_PRD.md`；举报处于待分诊或调查中 | 具有举报处理权限的运营人员或平台管理员 | 被举报房源所属城市 | 事件后 60 秒内刷新；待分诊 4 小时、调查 24 小时 SLA | `report_id + 当前处理阶段` | 房源举报详情；完成分诊、合并、结案或转审核后更新，结案时完成 |
| 待分配线索 | `04_客户线索/01_线索池_PRD.md`；Lead 阶段为待分配 | 销售主管、具有分配权限的运营人员或平台管理员 | 授权城市或所属团队服务范围内的未分配 Lead | 事件实时创建、列表 60 秒刷新；创建后 2 小时 SLA | `lead_id + 待分配版本` | 线索池/详情分配抽屉；分配、合并或进入已流失后完成 |

| 下次跟进提醒 | `04_客户线索/04_跟进记录_PRD.md`；当前有效的 `next_follow_up_at` 到期 | 当前负责人；所属团队主管只读可见 | 经纪人仅本人，主管仅本团队 | 到期前 24 小时生成；每 15 分钟扫描，到期后为超时 | `lead_id + next_follow_up_at + 负责人ID` | 我的客户详情的新增跟进抽屉；提交新跟进、改期或转派后关闭旧任务 |

| 经纪人资质/启停待处理 | `05_商户经纪/02_经纪人管理_PRD.md`；身份/任职核验待处理，或停用前存在需转移的客户 | 具有人员管理权限的运营人员、销售主管或平台管理员 | 所属商户、团队和服务城市 | 状态事件后 60 秒内刷新；资质核验 24 小时、停用转移 8 小时 SLA | `broker_id + 资质或停用流程版本` | 经纪人管理详情；核验完成或客户转移完成后自动完成 |
| 房源发布状态维护 | `03_房源管理/01_房源列表_PRD.md`；上架房源 `updated_at` 超过 30 天，或已出租/下架申请待确认 | 房源维护运营、具维护权限的提交商户；主管只读团队范围 | 房源所属城市、商户及角色数据范围 | 每日 00:10 扫描；超过 30 天即超时；状态申请事件 60 秒内刷新 | `listing_id + 最后更新时间或发布状态版本` | 房源详情编辑/发布处理页；有效维护、确认下架或标记已出租后完成 |

### 7.2 列表字段

| 字段 | 展示规则 |
| --- | --- |
| 任务编号 | 全局唯一，支持复制 |
| 待办类型/标题 | 类型图标+可读标题；点击进入任务注册表规定的来源详情 |
| 业务对象 | 对象类型、名称、不可变内部 ID；对象失效显示“对象已失效” |
| 来源/创建时间 | 明示权威来源模块和来源 ID；北京时间 `YYYY-MM-DD HH:mm` |
| 优先级 | 高红、中橙、低灰；颜色同时配文字 |
| 负责人 | 姓名+团队；无负责人显示“待领取” |
| 截止时间 | 到分钟；已逾期显示逾期时长 |
| 状态 | 待处理、处理中、已完成、已取消 |
| 未读状态 | 当前用户首次看到新建或转派任务前为未读；打开详情即置已读，不改变业务状态 |
| 操作 | 去处理、领取、转派、催办；按权限与状态显示 |

默认排序为：已逾期优先，其次优先级高到低，再按截止时间升序、创建时间升序。支持按截止时间和创建时间切换排序。

## 8. 详情及表单

点击任务编号打开只读抽屉，包含任务信息、权威来源和来源 ID、来源版本、触发规则、SLA、负责人变化、处理记录、未读状态和通知记录。转派弹窗必填新负责人和转派原因（2–200 字），候选人仅包含同数据范围且具备来源页面操作权限的启用账号。领取操作需二次确认；催办弹窗可补充 200 字以内说明，10 分钟内同任务不可重复催办。

## 9. 核心操作与流程

1. 来源业务创建待办，系统计算负责人、优先级与截止时间。
2. 无负责人的任务由有权限用户领取，状态从待处理变为处理中。
3. “去处理”跳转来源页面；来源动作成功后通过业务事件自动完成待办，不允许仅在待办页手工完成。
4. 转派成功后原负责人失去处理入口，新负责人收到站内通知；完整转派链写入日志。
5. 来源对象撤回、删除或进入不再需要处理的终态时，待办自动取消并记录原因。

批量领取和批量转派上限 50 条，仅允许同类型、同城市且状态可操作的任务；结果按成功与失败逐条返回。

## 10. 状态与业务规则

状态流转：待处理 → 处理中 → 已完成；待处理/处理中可因来源失效进入已取消。已完成和已取消为终态。逾期是计算属性：当前时间晚于截止时间且状态非终态。截止时间由任务注册表来源规则生成，人工转派不得自动延期。高优先级不等同于逾期，两者可同时存在；未读状态是用户通知属性，不参与业务状态流转。

## 11. 异常与边界

- 加载中使用固定高度骨架；无数据说明当前筛选下无待办并提供清空筛选。
- 无页面权限展示申请权限提示，不返回任何数量。
- 来源对象失效时禁止处理，允许查看失效原因并刷新任务状态。
- 两人同时领取时，以服务端首次成功为准，后者提示负责人已变化并刷新该行。
- 转派对象在提交前被停用时，提交失败并重新加载候选人。
- 接口失败保留筛选和已选项，提供重试；不得以本地状态假定处理成功。

## 12. 数据与指标

数据来自统一待办表，来源业务通过幂等事件写入。汇总卡按当前用户数据权限统计，去重键为待办内部 ID。今日到期指截止时间位于北京时间当日 00:00:00–23:59:59；已逾期不包含终态；高优先级包含所有非终态高优先级任务。列表实时查询，汇总卡缓存不超过 60 秒，手动刷新绕过缓存。导出不属于本页 MVP。

## 13. 埋点、通知与审计

埋点：页面访问、汇总卡切换、筛选查询、去处理、领取、转派、催办、操作失败。领取、转派、取消和完成记录操作人、前后负责人、来源对象、请求 ID、北京时间和结果。新分配、被转派、截止前 24 小时、逾期时发送站内通知；相同任务同类提醒每日最多一次。

## 14. 验收标准

- 正常：不同来源待办可按统一规则展示，处理来源业务后 60 秒内转为已完成。
- 正常：默认排序严格遵循逾期、优先级、截止时间、创建时间顺序。
- 权限：用户看不到权限范围外的任务、汇总数量和敏感字段，直接访问待办 ID 返回无权限。
- 权限：转派候选人不包含无来源操作权限、跨数据范围或停用账号。
- 异常：并发领取只允许一人成功，失败方得到明确反馈且不覆盖负责人。
- 异常：来源对象失效、接口超时和部分批量失败均可恢复，不产生重复任务。
- 数据：汇总卡、筛选结果与来源页面相同条件下的数据一致，时间均按北京时间计算。

## 15. 依赖与风险

依赖各业务模块提供幂等事件、来源详情深链、负责人和终态定义；依赖账号权限、团队和站内通知服务。主要风险是来源模块状态口径不一致导致任务无法自动闭环，因此每类待办上线前必须登记唯一来源事件、完成事件、取消事件与 SLA。MVP 不支持用户自定义待办规则。
