# 商办租赁平台后台管理系统 MVP PRD

版本：V1.0  
日期：2026-07-17  
适用对象：产品、设计、研发、测试、运营  
产品定位：面向商业办公租赁平台的后台管理系统 MVP，用于支撑楼盘房源供给、客户线索承接、经纪人跟进与基础运营管理。

---

## 1. 文档目标

本文档定义商办租赁平台后台管理系统 MVP 版本的产品范围、角色权限、功能模块、核心页面、交互逻辑、状态流转、字段规则和验收标准。

MVP 的目标不是一次性覆盖完整商办交易平台，而是优先打通以下业务闭环：

1. 楼盘与房源可录入、审核、上架和维护。
2. 客户线索可进入后台、分配给经纪人并持续跟进。
3. 经纪人可记录客户跟进过程、推荐房源和客户意向变化。
4. 管理员和运营人员可看到基础经营数据和待办任务。

---

## 2. 背景与问题

商业办公租赁平台的核心价值在于连接企业租户与办公空间供给方。与普通信息发布平台不同，商办租赁存在以下特点：

- 房源信息复杂，涉及楼盘、单元、面积、租金、物业、装修、工位、可租日期等多维信息。
- 房源真实性要求高，虚假价格、已租未下架、图片不符会严重影响用户体验。
- 客户决策周期长，需要顾问反复沟通、确认需求和推荐房源。
- MVP 阶段先记录线上线索承接和跟进过程，线下带看和成交履约暂不纳入后台闭环。
- 平台运营需要知道线索从哪里来、谁在跟进、是否及时响应、是否有效。

因此，后台 MVP 必须优先解决“供给可信”和“线索转化可管理”两个问题。

## 3. 产品目标

### 3.1 业务目标

- 支撑平台运营团队维护核心城市的楼盘与房源数据。
- 支撑经纪人完成客户跟进、需求确认和房源推荐。
- 支撑管理人员查看线索、房源和跟进效率的基础数据。
- 降低虚假房源、重复线索、超时未跟进等运营风险。

### 3.2 用户目标

- 运营人员可以高效审核和维护房源。
- 经纪人可以清楚知道“今天要跟谁、客户需求是什么、推荐过哪些房源”。
- 销售主管可以分配线索、查看团队跟进状态和客户阶段。
- 平台管理员可以配置账号、角色、城市、商圈等基础数据。

### 3.3 成功指标

MVP 上线后建议追踪：

- 房源审核平均处理时长。
- 上架房源信息完整率。
- 线索首次跟进及时率。
- 线索有效率。
- 线索推荐房源率。
- 超过 24 小时未跟进线索数量。
- 已租未下架房源数量。

## 4. 产品范围

### 4.1 MVP 包含范围

| 模块 | 是否包含 | 说明 |
|---|---:|---|

- 预约带看管理。

- 多端商户开放平台。
- 复杂 CMS 专题搭建器。

## 5. 用户角色与权限

### 5.1 角色定义

| 角色 | 核心职责 |
|---|---|
| 平台管理员 | 管理账号、角色、城市、商圈、系统配置，拥有全局权限 |
| 运营人员 | 管理楼盘、房源、审核、内容状态和基础数据 |
| 销售主管 | 管理团队经纪人、分配线索、查看团队跟进状态 |
| 经纪人 | 跟进客户、确认需求、推荐房源、维护客户阶段 |
| 客服 | 新增线索、补充客户资料、处理基础咨询 |

### 5.2 权限规则

权限分为菜单权限、操作权限和数据权限。

菜单权限示例：

- 是否可访问房源管理。
- 是否可访问系统设置。

操作权限示例：

- 新增、编辑、删除、导入、导出、审核、上下架、分配、转派。

数据权限示例：

- 平台管理员可查看所有城市数据。
- 城市运营只能查看授权城市数据。
- 销售主管可查看所属团队线索。
- 经纪人只能查看分配给自己的线索和客户跟进记录。

## 7. 核心业务流程

### 7.1 房源上架流程

```text
运营/商户创建楼盘
→ 创建房源
→ 填写房源信息
→ 保存草稿或提交审核
→ 运营审核
→ 审核通过
→ 房源上架
→ 前台展示
→ 定期维护或下架
```

异常流程：

- 信息缺失：提交审核时拦截并提示缺失字段。
- 审核驳回：房源进入“已驳回”，创建人可修改后重新提交。
- 已出租：运营或经纪人可申请标记为“已出租”，通过后自动下架。
- 长期未更新：超过 30 天未更新的房源进入“待复核”提醒。

## 8. 功能需求

## 8.1 登录与账号体系

### 8.1.1 登录

用户通过手机号/账号 + 密码登录后台。

页面元素：

- 账号输入框。
- 密码输入框。
- 登录按钮。
- 忘记密码入口。

交互规则：

- 账号或密码为空时，点击登录后在输入框下方提示。
- 密码连续错误 5 次，账号锁定 15 分钟。
- 登录成功后跳转工作台。
- 登录账号被停用时，提示“账号已停用，请联系管理员”。

### 8.1.2 账号管理

功能：

- 新增账号。
- 编辑账号。
- 启用/停用账号。
- 重置密码。
- 分配角色。
- 绑定组织或团队。

关键字段：

| 字段 | 必填 | 说明 |
|---|---:|---|
| 姓名 | 是 | 后台展示名称 |
| 手机号 | 是 | 唯一 |
| 登录账号 | 是 | 唯一 |
| 角色 | 是 | 可多选 |
| 所属城市 | 否 | 控制数据范围 |
| 所属团队 | 否 | 经纪人和主管必填 |
| 状态 | 是 | 启用/停用 |

验收标准：

- 停用账号不能登录。
- 手机号和登录账号不能重复。
- 修改角色后重新登录生效。

## 8.2 工作台

### 8.2.1 页面目标

让不同角色进入后台后立即看到待办任务、关键数据和快捷入口。

### 8.2.2 页面布局

页面包含：

- 顶部数据卡片。
- 我的待办。
- 今日任务。
- 快捷入口。
- 关键趋势图。

### 8.2.3 数据卡片

平台管理员/运营人员可见：

- 待审核房源数。
- 今日新增房源数。
- 今日新增线索数。
- 今日新增跟进数。
- 超时未跟进线索数。

经纪人可见：

- 我的新线索。
- 今日待跟进客户。
- 超时未跟进线索。
- 本月推荐房源次数。

销售主管可见：

- 团队新增线索。
- 团队待分配线索。
- 团队今日跟进数。
- 团队有效线索数。
- 超时未跟进线索。

### 8.2.4 交互规则

- 点击数据卡片跳转对应列表，并自动带入筛选条件。
- 待办任务按紧急程度排序：超时线索 > 待分配线索 > 待审核房源 > 今日跟进提醒。
- 数据权限按当前用户角色过滤。
- 无数据时展示空状态和快捷创建入口。

- 编辑楼盘。

列表字段：

### 8.3.2 新增/编辑楼盘

基础字段：

| 字段 | 必填 | 规则 |
|---|---:|---|
| 楼盘名称 | 是 | 同城市下不可重复 |
| 城市 | 是 | 从城市配置选择 |
| 区域 | 是 | 根据城市联动 |
| 商圈 | 是 | 根据区域联动 |
| 详细地址 | 是 | 文本输入 |
| 经纬度 | 否 | 支持地图选点或手动输入 |
| 楼盘等级 | 是 | 字典配置 |
| 竣工时间 | 否 | 年月 |
| 总楼层 | 否 | 数字 |
| 物业公司 | 否 | 文本 |
| 物业费 | 否 | 数字，单位元/㎡/月 |
| 停车位 | 否 | 数字 |
| 楼盘图片 | 否 | 最多 20 张 |

交互规则：

- 选择城市后，区域下拉项联动刷新。
- 选择区域后，商圈下拉项联动刷新。
- 楼盘名称重复时，保存前提示“当前城市已存在同名楼盘”。
- 未填写必填字段时，保存按钮可点击，但提交后定位到第一个错误字段。
- 停用楼盘前，如果存在上架房源，弹窗提示“该楼盘存在上架房源，停用后前台不可展示，是否继续”。

验收标准：

- 楼盘保存后可在房源新增页被选择。
- 停用楼盘后，不能新增该楼盘下的房源。

功能：

- 查询房源。
- 新增房源。
- 编辑房源。
- 提交审核。

筛选项：

- 房源状态。

- 面积区间。
- 租金区间。
- 更新时间。

列表字段：

| 字段 | 说明 |
|---|---|
| 房源标题 | 点击进入详情 |
| 所属楼盘 | 展示楼盘名称 |
| 城市/区域/商圈 | 支持筛选 |
| 面积 | 单位㎡ |
| 租金 | 元/㎡/天或元/月 |
| 装修状态 | 毛坯/简装/精装/带家具 |
| 可租日期 | 日期 |
| 状态 | 草稿/待审核/已上架/已下架/已出租/已驳回 |
| 信息完整度 | 百分比 |
| 更新时间 | 最近编辑时间 |
| 操作 | 编辑、提交审核、上架、下架、标记已出租 |

### 8.4.2 新增/编辑房源

基础字段：

| 字段 | 必填 | 规则 |
|---|---:|---|
| 所属楼盘 | 是 | 只能选择启用楼盘 |
| 房源标题 | 是 | 默认可由楼盘+面积+装修生成 |
| 房源类型 | 是 | 写字楼/联合办公/园区/商铺 |
| 面积 | 是 | 大于 0 |
| 租金单价 | 是 | 大于 0 |
| 租金单位 | 是 | 元/㎡/天、元/月 |
| 物业费 | 否 | 大于等于 0 |
| 楼层 | 否 | 文本，支持“中区/12层” |
| 工位数 | 否 | 正整数 |
| 装修状态 | 是 | 字典选择 |
| 可租日期 | 是 | 日期 |
| 最短租期 | 否 | 月数 |
| 付款方式 | 否 | 押一付三等 |
| 联系人 | 是 | 选择经纪人或商户联系人 |
| 房源图片 | 是 | 至少 3 张 |
| 户型图 | 否 | 最多 5 张 |
| 标签 | 否 | 多选 |
| 房源描述 | 否 | 富文本或长文本 |

按钮：

- 保存草稿。
- 提交审核。
- 取消。

交互规则：

- 保存草稿只校验楼盘、标题、城市等基础字段。
- 提交审核校验所有必填字段和图片数量。
- 面积或租金修改后记录价格变更日志。
- 图片上传支持排序，第一张为封面图。
- 删除图片时二次确认。
- 所属楼盘变更后，城市、区域、商圈自动跟随楼盘更新。

### 8.4.3 房源审核

审核对象：

- 新增房源。
- 已上架房源的核心字段变更。
- 已出租标记。

审核页面字段：

- 房源基础信息。
- 图片信息。
- 价格信息。
- 联系人。
- 提交人。
- 提交时间。
- 历史审核记录。

审核操作：

- 通过。
- 驳回。

交互规则：

- 点击通过后，房源进入“已上架”或“待上架”，由配置决定。MVP 默认通过后直接上架。
- 点击驳回必须填写驳回原因。
- 驳回后创建人收到待办提醒。
- 审核中房源不可再次编辑，如需编辑必须撤回审核。

状态流转：

```text
草稿
→ 待审核
→ 已上架
→ 已下架
→ 已出租

待审核
→ 已驳回
→ 草稿
→ 待审核
```

验收标准：

- 未审核通过的房源不能前台展示。
- 已下架房源不能被跟进记录关联为推荐房源。
- 已出租房源自动下架。

| 入驻时间 | 否 | 日期 |
| 租期 | 否 | 月数 |
| 特殊要求 | 否 | 长文本 |
| 线索来源 | 是 | 字典选择 |

交互规则：

- 输入手机号后自动查重。
- 若手机号已存在客户，展示弹窗：已有客户、负责人、当前阶段、最近跟进时间。
- 查重弹窗操作：合并到已有客户、创建新需求、取消。
- 新增线索后默认进入“新线索”。
- 如果当前用户是经纪人，新增线索默认负责人为自己。
- 如果当前用户是客服或运营，新增线索默认进入未分配状态。

### 8.5.3 线索分配

操作入口：

- 线索列表单条分配。
- 线索列表批量分配。
- 线索详情分配。

交互规则：

- 仅销售主管和管理员可分配。
- 分配时必须选择经纪人。
- 只能选择与线索城市匹配的经纪人。
- 分配后生成系统跟进记录：“线索由 A 分配给 B”。
- 已流失、无效状态的线索不能分配。

### 8.5.4 跟进记录

跟进方式：

- 电话。
- 微信。
- 面谈。
- 短信。
- 其他。

字段：

| 字段 | 必填 | 说明 |
|---|---:|---|
| 跟进方式 | 是 | 下拉选择 |
| 跟进结果 | 是 | 已联系/未接通/需求变化/已推荐/待下次跟进/无效等 |
| 下次跟进时间 | 否 | 日期时间 |
| 跟进内容 | 是 | 最少 5 个字 |
| 关联房源 | 否 | 可多选 |

交互规则：

- 提交跟进后更新线索最后跟进时间。
- 跟进结果为“已推荐”时，必须关联至少一个房源。
- 跟进结果为“无效”时，必须选择无效原因。
- 设置下次跟进时间后，工作台生成提醒。

线索阶段流转：

```text
新线索
→ 已分配
→ 已联系
→ 需求确认
→ 已推荐

任意有效跟进状态
→ 已流失

任意有效跟进状态
→ 无效
```

验收标准：

- 经纪人只能跟进自己的线索。
- 每次跟进必须保留历史记录，不允许物理删除。
- 超过 4 小时未首次跟进的线索在工作台标红提醒。

## 9. 全局交互规则

### 9.1 列表页通用规则

- 支持筛选、搜索、重置。
- 支持分页，默认每页 20 条。
- 支持按创建时间、更新时间排序。
- 支持列表字段根据角色控制展示。
- 无数据时展示空状态。
- 加载中展示骨架屏或加载状态。
- 查询失败展示错误提示和重试按钮。

### 9.2 表单页通用规则

- 必填字段用明显标识。
- 提交失败时定位到第一个错误字段。
- 退出未保存表单时弹窗确认。
- 保存成功后提示成功，并返回列表或停留当前页，由页面场景决定。
- 长表单建议分组展示：基础信息、价格信息、媒体信息、联系人、备注。

### 9.3 弹窗通用规则

- 危险操作必须二次确认。
- 驳回、取消、流失、无效等负向操作必须填写原因。
- 弹窗提交期间按钮置为 loading，防止重复提交。

### 9.4 数据脱敏规则

- 客户手机号默认展示为 `138****1234`。
- 经纪人本人、主管、管理员可查看完整手机号。
- 导出客户数据需要单独权限。

### 9.5 导入导出规则

MVP 支持线索导入，暂不强制支持所有模块导入。

线索导入规则：

- 支持 xlsx。
- 必填列：客户姓名、手机号、意向城市、线索来源。
- 导入前先预校验。
- 错误数据可下载错误报告。
- 重复手机号默认不导入，报告中提示重复原因。

## 12. 非功能需求

### 12.1 性能

- 常规列表页查询响应时间不超过 2 秒。
- 复杂筛选查询响应时间不超过 5 秒。
- 图片上传单张不超过 10MB。
- 后台支持至少 100 个并发管理用户。

### 12.2 安全

- 后台必须登录后访问。
- 接口必须校验用户权限和数据权限。
- 密码加密存储。
- 敏感操作写入日志。
- 客户手机号按权限脱敏。

### 12.3 可用性

- 表单错误提示明确到字段。
- 所有关键操作有成功或失败反馈。
- 审核、驳回、流失、无效等动作可追溯。
- 页面刷新后筛选条件可保留，至少在当前会话内保留。

## 13. MVP 验收标准

### 13.1 房源闭环

- 运营可创建楼盘。
- 运营可创建房源并提交审核。
- 审核人员可通过或驳回房源。
- 通过后的房源进入已上架状态。
- 已上架房源可被跟进记录关联为推荐房源。
- 已下架或已出租房源不能被新增推荐关联。

### 13.2 线索闭环

- 客服或运营可创建线索。
- 主管可将线索分配给经纪人。
- 经纪人可查看自己的线索。
- 经纪人可新增跟进记录。
- 跟进记录可改变线索阶段。
- 超时未跟进线索可在工作台提醒。

### 13.3 权限闭环

- 不同角色看到不同菜单。
- 经纪人不能查看其他经纪人的客户。
- 城市运营不能查看未授权城市数据。
- 关键操作均有操作日志。

## 15. 风险与待确认事项

### 15.1 主要风险

- 房源字段过多，录入成本高，可能影响运营效率。
- 线索来源多，去重规则如果过严可能误合并客户。
- MVP 暂不覆盖带看与成交，无法完整衡量线索到交易的最终转化。
- MVP 暂不接电话和 IM，跟进记录依赖经纪人手动录入。

### 15.2 当前默认假设

- MVP 以平台内部后台为主，不开放完整商户自助后台。
- MVP 不包含预约带看、成交报备、合同归档和财务结算。
- MVP 中前台展示由房源状态控制，但本文档不展开前台产品。
- MVP 中城市、区域、商圈由平台管理员维护。
